# Datagrok Coding Standards (common rules)

Canonical rules that apply to **both** the open `public/` repo (JS/TS plugins, libraries, tools) and the private `core/` repo (Dart client/server). Dart/core-specific extensions and the `code-reviewer` agent's operating procedure live in `core/docs/CODING_STANDARDS.md`.

## 1. Code minimization (highest-priority — most reviews fail here)

- **No speculative abstraction.** Three similar lines beats a premature helper. Don't add new `*Helper`/`*Util`/`*Manager` classes to host one or two callers.
- **No backwards-compat shims, feature flags, or `// removed` comments** unless you are shipping a public API change. Renamed internals should just be renamed.
- **No defensive validation of internal inputs.** Validation belongs at system boundaries (HTTP handlers, user input, external APIs). Don't write `if (x == null) throw …` on values produced by trusted internal code.
- **No try/catch that just rethrows or logs-and-rethrows.** Either handle the error meaningfully or let it propagate.
- **No half-implementations** (TODOs, stubbed methods, unused parameters added "for later") unless explicitly scoped.
- **Comments.** Do not write excessive comments. Don't restate what the code does. Keep only comments that explain a *non-obvious why* (hidden constraint, workaround, surprising invariant). Strip references to tickets, callers, or "added for X" — those rot.

## 2. Reuse — don't duplicate what already exists

Before adding a new helper, search the repo for prior art (`Grep` for the function the change re-implements; e.g. if the change adds `formatBytes`, grep for `formatBytes\|format_bytes\|humanSize`).

For JS/TS plugin code (`public/packages/`, `public/libraries/`):
- **Server calls:** `grok.dapi.*` (see `public/js-api/src/dapi.ts`). Use `grok.dapi.fetchProxy()` for external URLs. Never use raw `fetch()` against Datagrok endpoints.
- **UI primitives:** `ui.button`, `ui.icons.*`, `ui.input.*`, `ui.divV/divH`, `ui.tableFromMap`, `ui.dialog`, `DG.InputBase`, etc. Don't hand-roll `<button>`/`<input>` DOM construction when a `ui.*` helper exists.
- **Data:** prefer `DG.DataFrame`, `DG.Column`, `BitSet` operations over manual array juggling.

Dart/core reuse rules (typed `dapi.*` clients, `ddt`, `d4`, `GrokService.*`, `safeCallAsync`) live in `core/docs/CODING_STANDARDS.md`.

## 3. API call rules (hard rules — these are bugs, not style)

- **JS/TS plugins** MUST NOT call raw `fetch(` against Datagrok server URLs. Use `grok.dapi.*` for server calls, `grok.dapi.fetchProxy()` for external URLs.

Dart client (no raw `HttpRequest`/`http.get()`) and Dart server (`GrokService.*` only) rules live in `core/docs/CODING_STANDARDS.md`.

Reference: `core/shared/grok_shared/lib/src/http_client/CLAUDE.md` for the full Client ↔ Router ↔ Service mapping.

## 4. UI / CSS standards

### Design tokens
Defined in `core/client/xamgle/web/datagrok.css` `:root`. Always use these variables instead of hardcoding colors. Hardcoded colors (`#fff`, `rgb(…)`, `#1976d2`, etc.) are an issue.

- Colors: `--grey-1`..`--grey-6`, `--blue-1`, `--blue-2`, `--red-1`/`--red-3`/`--red-4`, `--green-1`..`--green-3`, `--steel-1`..`--steel-5`
- Semantic: `--text-color`, `--text-color-light`, `--success`, `--failure`

### Class-name prefixes
New classes must follow the convention for the area:
- `d4-` — D4 visualization framework (viewers, widgets)
- `ui-` — Xamgle UI components (inputs, panels, layout)
- `grok-` — Application-level features (views, entity rendering)
- `<package-name>-` — JS/TS packages (e.g., `power-pack-`, `hit-triage-`)

### Styling patterns
- Dart client uses both inline `.style.` (dynamic layout) and CSS classes (static presentation).
- JS/TS packages: CSS files imported in `.ts` files (bundled via webpack style-loader), plus inline `.style.`.
- **No SASS/SCSS** — plain CSS only. Don't add `.scss`/`.sass` files.
- **Inline styles** are acceptable for *dynamic* layout (computed sizes, runtime show/hide). Static presentation belongs in CSS classes — don't write `style.color = '#…'` for static styling.

## 5. JS/TS style

- 2-space indentation
- Single quotes for strings
- Semicolons required
- Windows line endings (CRLF)
- TypeScript strict mode
- File naming: kebab-case (`my-component.ts`)
- Package folder naming: PascalCase (`PowerGrid`, `BiostructureViewer`)
- Package name in package.json: kebab-case with scope (`@datagrok/power-grid`)
- Semantic versioning: `X.Y.Z` or `X.Y.Z-rc` or `X.Y.Z-rc.N`
- Never edit files ending in `.g.ts` or `.api.g.ts` — these are auto-generated by `grok api`

## 6. Testing

See `core/docs/platform/TESTING.md` for the full guide. Key rules:

- **Real server/DB, not mocks.** Prefer real server/DB calls over mocks. Never mock `grok.dapi.*`, the HTTP layer, or the database in integration tests.
- **New public JS API surface MUST have both** an `ApiTests` integration test (`public/packages/ApiTests/`) and an `ApiSamples` entry (`public/packages/ApiSamples/scripts/`).
- **Always clean up server-side state** created during tests (`tearDownAll` / `finally`).
- Use `/grok-add-api-coverage` to scaffold the JS API test + sample.

## 7. Plugin changelog rule

When committing a sizable feature or bug fix to a plugin under `public/packages/`, add a one-line entry to the `## v.next` section at the top of that plugin's `CHANGELOG.md`. Create the section if it doesn't exist yet. Trivial changes (typos, formatting, dependency bumps) are exempt.

Format:

```markdown
# PackageName changelog

## v.next

* Component: Added new feature description
* GROK-12345: Fixed something

## 1.17.2 (2026-03-23)
...
```

- One bullet per logical change, flat list (no subsections).
- Start with ticket ID if available (`GROK-NNNNN:` or `[#NNN](url):`).
- Use past-tense verbs: Added, Fixed, Improved, Introduced, Implemented.
- Skip trivial changes (typos, formatting, dependency bumps).
- When the plugin is published, `/plugin-changelog` converts `v.next` to a versioned entry.

## 8. Cross-platform safety (Windows & Unix)

- **Never create files named** `NUL`, `CON`, `PRN`, `AUX`, `COM1`–`COM9`, `LPT1`–`LPT9` — Windows reserves these.
- **Do not redirect output to `nul`** or create placeholder files using reserved device names on Windows. Files named `nul` may be untracked artifacts of a prior mistake — call them out.
- When discarding output:
  - Unix/macOS: use `/dev/null`
  - PowerShell: use `$null`
- Prefer platform-aware or conditional commands in scripts. Flag platform-specific paths/commands without a conditional fallback in scripts meant to run cross-platform.

These rules apply to **all generated commands, scripts, and instructions** in this repository.

## 9. Security and correctness (always-on)

- **Injection:** command injection, SQL injection, XSS via `innerHTML`, unescaped user input in shell commands.
- **Secrets:** no secrets/credentials in code or test fixtures.
- **Standard bug classes:** off-by-one, null-deref, `async` without `await`, swallowed exceptions, race conditions on shared state.
- **Wrong concurrency primitives:** `setState` on disposed widgets, listeners not removed, etc.
