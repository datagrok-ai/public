# Ревью плана `LambdaZStrategy.minSpanRatio`

> **Что это:** инженерное ревью спецификации [`sci-comp-min-span-ratio.md`](./sci-comp-min-span-ratio.md).
> **Метод:** план сверен с фактическим кодом, фикстурами и reference-suite, а не только прочитан.
> **Дата:** 2026-06-03.

---

## TL;DR

- Адресация в плане (файлы, строки, точки вставки) — **точна**, патч скомпилируется.
- **Главная проблема:** план неверно моделирует семантику PKNCA `min.span.ratio`. В PKNCA это **диагностический флаг** на выбранном фите, а **не** жёсткий фильтр кандидатов, отбрасывающий фит и возвращающий `null`. Доказательство — в самих фикстурах (см. §2.1).
- Из-за этого шаг 3 (усилить reference-suite значением `minSpanRatio: 2`) и шаг 5 («все тесты зелёные, no drift») **взаимно противоречивы** — в текущем виде верификация недостижима.
- **Breaking changes:** на уровне типов и для существующих вызовов с дефолтами — **НЕ breaking** (обоснован minor-релиз). Реальные риски — не в совместимости API, а в семантике и в возможной смене дефолта на `2`.

## 2. Риски (по убыванию серьёзности)

### 🔴 2.1. Неверная семантика PKNCA → reference-suite сломается

План (строка 90) утверждает: *«фикстуры совпадают, потому что их авто-фиты и так удовлетворяют `span/halfLife ≥ 2`»*. **Это неверно.**

Проверка по провенансу фикстуры **Theoph, subject 1** ([01_theoph.json:37-45](./src/nca/__tests__/fixtures/01_theoph.json#L37)):

```
lambda_z_time_first = 9.05
lambda_z_time_last  = 24.37   → span = 15.32
half_life           = 14.304
                    → span ratio = 15.32 / 14.304 = 1.07   (< 2!)
```

При этом PKNCA вернул валидный `lambda_z = 0.0485`, и фикстура его **ожидает**.

Вывод: в PKNCA `min.span.ratio` — это **флаг качества** на уже выбранном фите, а **не** критерий, отбрасывающий кандидата и дающий `NA`. Фикстуры сгенерированы с `min_span_ratio: 2`, но всё равно содержат фиты с ratio < 2.

**Последствия:**
- Реализация в плане — `continue` внутри цикла кандидатов, т.е. **жёсткий gate**, возвращающий `null` → `status: 'partial'`. Это **расходится с PKNCA**, на эквивалентность которому план опирается как на главную ценность.
- Шаги 3 и 5 несовместимы: включив `minSpanRatio: 2` на theoph subject 1, мы отбросим его окно (ratio 1.07) → выбор другого `lambda_z` или `null`/`NaN` → тест `relErr(lambdaZ, 0.0485)` ([reference-suite.test.ts:168](./src/nca/core/__tests__/reference-suite.test.ts#L168)) **упадёт**. «Verification green» в текущем виде недостижима.

### 🔴 2.2. Тест #3 «fallback to wider window» кодифицирует неверное поведение

PKNCA при непрохождении span-ratio **не** ищет более широкое окно — он берёт лучшее по adj-R² и флагует span. Per-candidate `continue` меняет **алгоритм выбора окна** (может выбрать не то окно, что PKNCA). Тест #3 закрепляет это расхождение как «правильное». Решение о семантике нужно принять **до** написания тестов.

### 🟡 2.3. `partial` теряет информацию о причине

Сейчас `lambdaZBestFit → null` означает «терминальную фазу оценить нельзя вообще». Если span-gate тоже отдаёт `null`, потребитель на уровне `ComputeResult` не отличит «фита нет» от «фит есть, но span короткий». PKNCA эти случаи различает (значение + флаг). Это бьёт прямо по заявленной цели плана (строка 34: *«documented, configurable consequence», а не «engine quietly decided»*) — получается наоборот. Раздел 8 плана («expose `spanRatio`») признаёт это, но выносит за рамки v1 — а именно это и нужно для нормального UX.

### 🟡 2.4. Расхождение дефолтов sci-comp ↔ nca-studio

`undefined` (off) безопасно. Но раздел 7 оставляет открытым выбор `2` для UI. Если nca-studio когда-нибудь поставит дефолт `2`, профили, сейчас дающие `lambda_z`, массово станут `partial` — видимая регрессия для пользователя. Нужно зафиксировать: дефолт `undefined`, gate только при явном вводе.

### 🟢 2.5. Мелочи
- Guard `&& fit.lambdaZ > 0` — **мёртвый код**: строкой выше ([lambda-z.ts:49](./src/nca/core/lambda-z.ts#L49)) уже стоит `if (fit.lambdaZ <= 0) continue;`. Рассуждение про `halfLife = +Infinity` в разделе 8 неактуально по той же причине.
- Раздел 6 говорит «release sci-comp», но это пакет монорепо — релиз идёт обычным publish-флоу; стоит сослаться на реальный процесс.

## 3. Breaking changes — вердикт

### Изменение публичного API / типов — НЕ breaking
- Опциональное поле `minSpanRatio?: number` аддитивно: существующий код, конструирующий `LambdaZStrategy`, компилируется без изменений (TS-совместимо).
- При `minSpanRatio === undefined` рантайм идентичен текущему → для потребителей, не задающих поле, **поведение не меняется**.
- По SemVer — **minor**, не major.

### Оговорки, при которых поведение всё же ломается
1. **Шаг 3 (reference-suite с `minSpanRatio: 2`)** в текущем виде **сломает тесты библиотеки** (см. §2.1). Это не breaking для потребителей API, но шаг нерабочий — «всё зелёное» из шага 5 недостижимо.
2. **Если дефолт станет `2`** в любом звене (sci-comp или nca-studio) — это **поведенческий breaking change** для пользователей (`ok → partial`). Пока дефолт `undefined` — не breaking.
3. **«PKNCA-equivalent» как контракт**: per-candidate gate даёт **другой** выбор окна, чем PKNCA — семантическое расхождение, хотя типовой контракт не нарушается.

**Итог:** на уровне типов и для существующих вызовов с дефолтами — **не breaking**, minor-релиз обоснован. Реальные риски лежат не в совместимости API, а в (а) неверной модели семантики PKNCA, из-за которой обещанная верификация недостижима, и (б) поведенческом сломе при смене дефолта на `2`.

## 4. Развилка, которую надо закрыть до реализации

Один вопрос меняет весь дизайн:

### Вариант A — PKNCA-faithful (рекомендую)
`minSpanRatio` — **диагностический флаг**. Выбор окна **не меняется**; `lambdaZBestFit` по-прежнему возвращает лучший фит, но добавляет `spanRatio` (и/или warning `LAMBDAZ_LOW_SPAN`) в `LambdaZResult` / provenance.
- ✅ Reference-suite остаётся зелёным **без** изменения ожиданий.
- ✅ Совпадает с тем, что реально делают фикстуры.
- ✅ Сохраняет информацию о причине (§2.3) — не схлопывает в `partial`.
- ✅ Действительно «PKNCA-equivalent».

### Вариант B — hard-gate (как в текущем плане)
Окно реально отбрасывается, при провале — `null` → `partial`.
- ⚠️ Reference-suite **нельзя** гонять с `minSpanRatio: 2` против существующих фикстур — нужны отдельные фикстуры/ожидания.
- ⚠️ Заявление «PKNCA-equivalent» придётся убрать/уточнить.
- ⚠️ Нужна доработка provenance, чтобы отличать причины `partial`.
