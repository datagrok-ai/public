# Phase 16 — UI Design Contract

> Visual and interaction contract for sample-level SPC tracking. Generated by gsd-ui-researcher; verified by gsd-ui-checker.

**Phase 16 has NO new component-library / styling decisions.** This package is a Datagrok plugin. UI is composed exclusively from Datagrok platform primitives. The UI-SPEC's job is to lock COPYWRITING, INTERACTION FLOW, EMPTY STATES, ERROR STATES, ARRANGEMENT, and AUDIENCE-AUDITED LANGUAGE — not to pick a component library.

**Audience pin (LOAD-BEARING, carried from Phase 15):** Cytokinetics demos may include biologists. Every reviewer-facing surface (dashboard banner, baseline-define modal copy, drill-down toasts, Pareto labels, rule-toggle labels) is biologist-readable.

**Banned jargon** (every occurrence is a checker failure): `DataFrame`, `tag`, `semType`, `viewer factory`, `ACL`, `helper`, `dispatcher`.
**Use instead:** "run", "analyzed file", "QC metric", "out-of-control point", "baseline", "table".

## Spacing Scale

Datagrok platform inherits its own theme spacing. Phase 16 uses Datagrok's standard `ui.divV` / `ui.divH` default flow — no custom CSS, no `style` overrides. Declared values are guidance for any explicit padding/margin used inside the Define-baseline modal:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gap inside the banner; inline checkbox label spacing |
| sm | 8px | Compact element spacing (checkbox row gap inside the Define-baseline modal) |
| md | 16px | Default element spacing (between groups of inputs in the modal) |
| lg | 24px | Section padding (banner inner padding; sidebar block separators) |
| xl | 32px | Layout gaps (between the 4 I-charts in the dashboard) |

Exceptions: none. Dock-managed panels (the 4 I-charts, the sidebar, the Pareto panel) are arranged by `grok.shell.tv.dockManager` and inherit platform-default gutter; do NOT override.

## Color

Datagrok platform owns the dominant + secondary palette. Phase 16 declares ONLY accent colors used for SPC status semantics:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | Platform default surface | Background, table surfaces (do not override) |
| Secondary (30%) | Platform default sidebar / dock-panel background | Sidebar, modal background (do not override) |
| Accent — pass | `#3CB371` (medium sea green) | I-chart point color when the run's status is `pass`; status badge in sidebar |
| Accent — flagged | `#F0AD4E` (amber) | I-chart point color when status is `flagged`; "Newer version" / sidebar warning text |
| Accent — out_of_control | `#D9534F` (red) | I-chart point color when status is `out_of_control`; banner border-left when no baseline locked |
| Formula line — CL (center line) | `#888888` (mid grey) | Center-line formula in I-chart and MR-chart; matches QC dashboard's MA=0 reference line precedent |
| Formula line — UCL / LCL | `#666666` (darker grey) | Upper / lower control limits in I-chart and MR-chart (1px dashed; platform `lineChart` formula-lines default) |

Accent reserved for:
- **Pass / flagged / out_of_control point coloring on the four I-charts** — and ONLY on the four I-charts. MR-chart points use platform default (no traffic-light coloring; MR is a derived signal, not a status).
- **Status badge in the sidebar** — `"Baseline locked: 2026-07-10 · 7 runs included / 1 excluded · status: pass"` — the trailing word picks one of the three accent colors.
- **Empty-state banner border-left** — when no baseline locked: red 4px left border on the banner block.
- **Pareto chart** — bars use platform default barChart palette. No traffic-light recoloring. (Pareto answers "which rule trips most often" — orthogonal to per-run status.)

Never use accent for: input borders, button backgrounds, chart axes, sidebar block borders, modal frame, hover states.

**Destructive color** (`#D9534F`): used ONLY for `out_of_control` status and for the "Rebuild baseline..." button's confirmation toast. No "delete" action exists in Phase 16 surface — rebuild OVERWRITES (D-04, deferred: undo is v1.5+).

### False-alarm-rate disclosure (LOAD-BEARING per Pitfall 5)

Live disclosure tooltip on the per-metric rule-toggle grid in the sidebar. The tooltip text rewrites itself as the user toggles rules ON/OFF for any metric. Computed against Nelson 1984 published false-alarm rates per rule, summed (with the standard independence-assumption caveat the literature notes).

| Rule subset enabled (for this metric) | Tooltip body |
|---|---|
| `nelson_1` only | `Expected false-alarm rate at this rule set: ~0.27%. (Rule 1 only — a point beyond 3σ.)` |
| `nelson_1` + `nelson_5` (DEFAULT) | `Expected false-alarm rate at this rule set: ~0.4%. (Rules 1 + 5 — a single point beyond 3σ, or 2 of 3 beyond 2σ.)` |
| `nelson_1` + `nelson_5` + `nelson_2` | `Enabling rules 1+5+2 raises the expected false-alarm rate to ~1.1%.` |
| All 8 rules enabled | `Expected false-alarm rate at this rule set: ~2.65%. Enabling all 8 rules will flag roughly 1 in 38 normal-operation runs.` |

Generic shape for any other subset (computed live):
> `Expected false-alarm rate at this rule set: ~<X>%. Enabling more rules makes the chart more sensitive but raises the chance of flagging a normal-operation run.`

## Interaction Flow — Locked Sequences

### 1. Compute SPC Status (`Proteomics → Analyze → Compute SPC Status`)

```
[user clicks menu]
  └─ menu handler in package.ts
      ├─ check: tv?.dataFrame exists → else warning + return
      ├─ check: spc_run_meta non-empty → else warning ("Open Annotate Experiment...")
      ├─ check: source !== 'spectronaut-candidates' → else hard refuse
      ├─ check: getGroups(df) !== null → else warning
      ├─ open TaskBarProgressIndicator("Computing SPC status...")
      ├─ computeSpcMetrics(df, groups, runMeta)
      ├─ evaluateNelsonRules(metrics, baseline, rules_enabled)
      ├─ setSpcStatus(df, metrics, ruleResult)        ← writes tags + spc_metrics_meta column
      ├─ appendRun(row) → AppData runs.csv             ← idempotent on (instrument_id, datetime)
      ├─ close progress indicator
      └─ shell.info success toast (see Copywriting → Success confirmations)
```

No dialog — immediate action. User feedback path: progress bar → success/warning toast.

### 2. SPC Dashboard first open (`Proteomics → Visualize → SPC Dashboard...`)

```
[user clicks menu]
  └─ check: runs.csv exists with at least 1 row → else empty-state banner
  └─ read userDataStorage('spc.last_picked_instrument')
  └─ build instrument list from runs.csv (deduped (instrument_id, has-baseline?) pairs)
  └─ open instrument picker (ui.dialog with choice input)
      └─ on OK:
          ├─ persist pick to userDataStorage
          ├─ grok.shell.addTableView(loadRuns(instrumentId))   ← sibling table view
          ├─ dockManager.dock(sidebar, LEFT, 0.25)
          ├─ check: baseline-<instrumentId>.json exists
          │   ├─ YES → render 4 I-charts + 4 MR-charts + Pareto (P2)
          │   └─ NO  → render empty-state banner inside the table view
          └─ wire I-chart point click → drill-down handler
```

### 3. Define baseline (from sidebar `[Define baseline...]` button)

```
[user clicks Define baseline]
  └─ open ui.dialog("Lock baseline for <instrument_id>")
      ├─ render scrollable checkbox list of all runs for this instrument
      │   ├─ default: first 7 runs checked
      │   └─ checkbox rows ordered by acquisition_datetime ascending
      ├─ render "Iterate 3σ outlier removal" toggle (default ON)
      ├─ render per-metric Nelson rule toggle grid (4 metrics × 8 rules)
      │   └─ default: rules 1 + 5 ON for every metric
      ├─ on OK:
      │   ├─ iterativeOutlierRemoval(checked_runs, max_iter=2)
      │   ├─ computeBaselineStats(remaining_runs)
      │   ├─ writeBaseline(instrumentId, baseline_json)
      │   ├─ refresh charts in place (no view re-open)
      │   └─ shell.info "Baseline locked..." toast
      └─ on Cancel: dialog closes, no state change
```

### 4. Rebuild baseline (from sidebar `[Rebuild baseline...]` button)

Same flow as Define baseline, with two diffs:
- Title: `Rebuild baseline for <instrument_id>`
- Modal pre-populated with the current `included_run_ids` checked + current `rules_enabled` grid
- On OK: overwrite baseline-<instrument_id>.json with destructive-confirmation copy (see Copywriting)

### 5. Drill-down (click flagged point on I-chart)

```
[user clicks I-chart point]
  └─ lineChart row-changed event fires
  └─ read runs.csv row at clicked index
  └─ if source_project_id non-null:
      ├─ grok.dapi.projects.find(id).then(p => p?.open())
      └─ silently opens; the project's own auto-dock surfaces take over
  └─ else: shell.warning (see Copywriting → Drill-down)
```

### 6. Rule toggle (live-update from sidebar)

```
[user toggles a checkbox in the per-metric rule grid]
  └─ no dialog, no confirm
  └─ update rules_enabled in-memory
  └─ writeBaseline(instrumentId, updated_baseline)  ← persists immediately
  └─ re-evaluate Nelson rules over all runs in chart
  └─ update I-chart point colors in place
  └─ recompute false-alarm-rate disclosure tooltip text
```

- LCL = `baseline.metrics.<metric>.mean - 3 × baseline.metrics.<metric>.sd` (1px dashed, darker grey)

- UCL = `D4 × mean(MR)` where D4 = 3.267 for n=2 (Wheeler constants, locked in research)
- CL = `mean(MR)` (1px solid, mid grey)

**Point coloring on I-charts ONLY:** read each run's `status` from runs.csv → map to accent palette above. MR-chart points use platform default (no traffic-light recoloring).
