Close Gap A from the current 13-UAT.md (Test 2 issue, severity major): after the user runs Proteomics | Analyze | Enrichment, the dot+bar charts feel "associated with the enrichment table and not with the table that has the volcano plot". The cross-link data-layer wiring is correct and is firing — but the visible effect (highlighted points) renders on a volcano the user is not looking at.

The previous gap-closure plan (13-07, Option B) co-docked a SECOND volcano on the enrichment view to give the cross-link a visible target there. The current UAT round (post-13-07) confirms the user STILL reports the same complaint verbatim. Option B was insufficient because:
  (a) The user's primary working view is the protein view (where they originally opened Proteomics | Visualize | Volcano Plot). They expect their cross-link target to be THEIR volcano, not a 0.4-ratio "Volcano (linked)" duplicate that opened sideways on a freshly-switched-to view.
  (b) Option B leaves the user on the enrichment view — they perceive the dot/bar charts as "the enrichment table's charts" because that is exactly where they are docked.
  (c) The user's verbatim phrasing — "enrichment charts are associated with the enrichment table and not with the table that has the volcano plot" — is a direct request for Option A.

This plan implements **Option A**: dock the Up/Down dot+bar viewers (and the single-direction fallback dot+bar pair) onto the **protein** TableView (the one that holds the user's volcano), and switch focus back to that view after the docking finishes. The merged enrichment DataFrame remains a separate TableView (accessible via tabs) so the merged grid + Direction column + smart-filter banner stay browseable, but the protein view is the surface the user lands on and is where the dot/bar charts live alongside the original volcano.

Removes the now-redundant 13-07 "Volcano (linked)" co-docked volcano from both the directional-split and single-direction fallback branches — the user's existing volcano on the protein view is the cross-link target, so a second volcano on a separate view adds noise.

Purpose: resolve the user's verbatim perception failure by putting the enrichment charts in the same field of view as the cross-link effect. The data-layer wire (`wireEnrichmentToVolcano` → `proteinDf.selection`) is preserved unchanged; only the dock target and the post-dock focus change.

Output: openEnrichmentVisualization signature stays the same (`(enrichDf, proteinDf) => DG.TableView`), but its dock-target resolution flips: the dot+bar viewers go on the protein TableView; the enrichment TableView keeps the merged grid + smart-filter banner only; `grok.shell.v` lands on the protein view at the end. <strategy_justification>
**Chosen: Option A** — dock the dot+bar viewers on the **protein** TableView (the one with the user's volcano); switch focus to that view after docking; remove the 13-07 "Volcano (linked)" co-dock.

**Why not Option B (current shipped 13-07 behavior):**
13-07 took Option B (co-dock proteinDf-bound volcano on enrichment view). The current UAT round (post-13-07) shows the user reports the identical verbatim complaint. The 13-07 SUMMARY noted human-UAT was needed to confirm the gap closed; this UAT is the answer — it did not. Specific reasons B fell short:
  - At ratio 0.4 on a view with 4 dot/bar viewers + merged grid + optional smart-filter banner, the linked volcano is too cramped to be a primary working surface
  - The user's mental model attaches "their volcano" to the table they imported. A second "Volcano (linked)" on a different view feels like a duplicate, not "their volcano"
  - The user still lands on the enrichment view by default — the dot/bar charts ARE docked there, fulfilling the "associated with the enrichment table" perception

**Why not Option C (status text / auto-bring-to-front only, keep B's layout):**
C is a partial fix — it adds awareness signals (status text on the enrichment side, or auto-bringing the protein view to front on row click) but the dot/bar charts STILL physically live on the enrichment view. The user's verbatim complaint is about that physical association, not about awareness. Option C also forces the user to mentally track two views per click (which one is the current?). Option A eliminates the question.

**Why Option A is safe to ship after 13-07:**
  - The wire-up data-layer contract is unchanged: `wireEnrichmentToVolcano` still mutates `proteinDf.selection`. The original volcano on the protein view already renders that selection — that's the Phase-9 cross-link's whole point.
  - Removing the 13-07 co-docked "Volcano (linked)" eliminates a viewer that was redundant under Option A (the original volcano IS the cross-link target).
  - The enrichment TableView still gets created by `addTableView(result.enrichmentDf)` — the merged enrichment grid is still browseable on its own view, so users wanting the term/FDR/Intersection table directly can switch to it.
  - The protein view gains 2 new docked viewers (Up dot + Up bar + Down dot + Down bar = 4 in directional split, OR 1 dot + 1 bar in single-direction fallback). This is the same number 13-07 placed on the enrichment view; the load is just relocated. Existing volcano on the protein view is preserved (no removal, no resize forced).

**Risk note on layout density:**
The protein view will now host: original volcano + 4 (or 2) dot/bar viewers + any QC/PCA/etc viewers if the user ran Show All Visualizations. To prevent crowding, dock the dot/bar viewers RIGHT of the volcano (consistent with the 13-07 enrichment-view layout idiom) at ratio 0.5, anchored to the protein TableView's grid (i.e. `null` anchorNode, RIGHT). DG.DockManager auto-handles re-flow when multiple viewers compete for the same edge.
</strategy_justification>
