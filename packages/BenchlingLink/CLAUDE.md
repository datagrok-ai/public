# BenchlingLink

Datagrok package that exposes [Benchling](https://www.benchling.com/) registry content (AA sequences, DNA sequences & oligos, molecules, mixtures, plates, assay results/runs, projects) as typed Datagrok functions and a tree-browser app under `Home | Browse | Apps | Chem | Benchling`.

## Purpose

- Wrap the Benchling REST API v2 (`https://benchling.com/api/v2/...`) as Datagrok package functions.
- Provide a "pick a resource in the tree → get the standard platform function preview" browsing UI.
- Provide `createX` counterparts for posting new entities back to Benchling.

## Critical: mock-backed state

**Every query/post function in `src/*Api.ts` currently returns mock data.** In each file the real `grok.dapi.fetchProxy(...)` call is present but commented out with a `'YOUR_BENCHLING_API_TOKEN'` placeholder, and the live code path returns a `mockXxx` fixture (or derives one from `files/benchling_mock_molecules.csv` for molecules). The UI, parameter wiring, and `dataFrameFromObjects` pipeline are real — only the HTTP hop is stubbed.

`src/credentials.ts` already defines `getApiKey()` reading the package credential parameter `apiKey`, but it is **not imported anywhere**. It is waiting to be wired in when the commented blocks are activated.

Before changing anything in this package, decide whether the task is meant to run against (a) mocks, (b) real Benchling, or (c) both — and, if (b)/(c), expect to uncomment the HTTP block and replace the token literal with `await getApiKey()`.

## ⚠️ Auth mechanism is UNVERIFIED

The current `src/credentials.ts` assumes a single static API key stored as the `apiKey` package credential and injected as `Authorization: Bearer <apiKey>` by every commented HTTP block. **This has never been tested against a live Benchling tenant and may not be sufficient.**

The swagger tells a different story: `POST /token` (around line 17482 of `swaggers/benchling_openapi.yaml`) describes **OAuth2 client-credentials auth** (`basicClientIdSecretAuth`) — a `clientId` + `clientSecret` pair exchanged for a short-lived access token that must be refreshed on expiry (`TokenResponse.expires_in`). Some tenants accept tenant-issued API tokens (the bearer flow assumed here), others require the OAuth2 handshake.

**Before activating real HTTP for any resource, treat the auth layer as an open question, not a solved one.** Specifically:

1. Confirm with the target tenant admin which scheme is enabled: static API token, OAuth2 client-credentials, or both.
2. If OAuth2 is required, `credentials.ts` must be rewritten end-to-end:
   - Package credential parameters become `clientId` + `clientSecret` (not `apiKey`).
   - Add a `POST /token` call with `application/x-www-form-urlencoded` body to mint an access token.
   - Cache the token with a TTL driven by `expires_in`, not forever.
   - Refresh proactively before expiry, and retry once on 401.
3. Revisit the module-level `apiKey` cache even in the static-bearer case — it holds the credential for the full page session, so credential rotations on the server take effect only after a reload; concurrent first callers race on `_package.getCredentials()`.
4. Update `getApiKey()`'s error messages, or replace the function entirely, so callers get actionable guidance when auth fails.

Do not assume the current `getApiKey()` shape is load-bearing. The whole auth mechanism should be investigated and re-implemented if required. The CLAUDE.md section on "Activating real HTTP for a resource" below assumes the static-bearer path only — treat that as a placeholder until auth has been verified end-to-end.

## Architecture

```text
src/
  package.ts           Entry: app, tree browser, all @grok.decorators.func wrappers
  <resource>Api.ts     One file per Benchling resource — interfaces + query/post + mock fixture
  utils.ts             dataFrameFromObjects + buildQueryString + translateParamName
  credentials.ts       getApiKey() — reads package-level `apiKey` credential (not yet wired)
  types.ts             Cross-resource shared types (e.g. UserSummary)
  package.g.ts         AUTO-GENERATED by `grok api` — do not edit
  package-api.ts       AUTO-GENERATED typed wrappers — do not edit
files/
  benchling_mock_molecules.csv   SMILES source for the Molecules mock fixture
swaggers/
  benchling_openapi.yaml         Upstream Benchling OpenAPI spec — reference when adding resources
```

### Per-resource file shape

Every `<resource>Api.ts` follows the same template:

1. `interface <Resource>` — field-level types for one entity.
2. `interface <Resource>PaginatedList` — `{ <resource>: [...], nextToken?: string }`, matching Benchling's pagination envelope.
3. `interface <Resource>QueryParams` — all supported GET filter params, using underscore suffixes (`_anyOf`, `_gt`, `_lte`, …) which `utils.translateParamName` converts to the swagger's dot form at request time.
4. `async function query<Resource>(params)` — **commented real call** + `return dataFrameFromObjects(mockXxx.<resource>);`.
5. `interface <Resource>CreateRequest` (singular create) or `interface <Resource>sBulkCreateRequest` (Assay results/runs, which require the bulk envelope).
6. `async function post<Resource>(body)` (or `post<Resource>s` for Assay) — **commented real call** + mock return.
7. `mockXxx: <Resource>PaginatedList` — 100-row fixture, typically generated in a `...Array.from({length: 100}, (_, i) => {...})` block.

### package.ts: what's load-bearing

- `benchlingLinkApp` (`@grok.decorators.app`) — static landing page (icon + description) under browse path `Chem`.
- `benchlingLinkAppTreeBrowser` (`@grok.decorators.appTreeBrowser`) — populates the tree with one node per resource. Each node's `value` is the `DG.Func`, and `onSelected` delegates rendering to the platform's default object-handler preview.
- `getX` functions (one per resource) — thin wrappers that build a `params` object and delegate to `query<Resource>(params)`.
- `createX` functions — parse any JSON-encoded inputs (`aliases`, `annotations`, `authorIds`, `customFields`, `fields`, `wells`, `ingredients`, `fieldValidation`, …), build a body, call the per-resource poster, and wrap the single returned object/response via `dataFrameFromObjects([result])`.

### How tree nodes and functions are paired

The tree browser knows the *platform function name* (`'BenchlingLink:getAASequences'`), not the TS symbol. The `@grok.decorators.func({name: 'Get AA Sequences'})` name is the *UI display name*; the platform function name comes from the static method name (camelCase). If you rename a `getX` method, update the matching `addFuncNode(label, 'BenchlingLink:<newName>')` call in `benchlingLinkAppTreeBrowser`.

### utils.ts: translation + DataFrame conversion

`utils.ts` exports three things used everywhere:

- **`translateParamName(key)`** — TS-friendly underscore suffixes → Benchling's dotted query-param names. Maps `_anyOf_caseSensitive` → `.anyOf.caseSensitive`, `_anyOf` → `.anyOf`, `_gte`/`_gt`/`_lte`/`_lt` → `.gte`/`.gt`/`.lte`/`.lt`, `_mol` → `.mol`, `_smiles` → `.smiles`. Ordinary keys pass through unchanged.
- **`buildQueryString(params)`** — URL-encodes only defined/non-empty params, drops empty arrays, applies `translateParamName` to every key, and comma-joins array values (Benchling uses `explode: false` for multi-value filters like `mentionedIn`, `mentions`).
- **`dataFrameFromObjects(objects)`** — flattens nested objects (`creator.name` → `"creator.name"` column), turns arrays of objects into comma-joined `id` / identifier CSVs, and infers column type from the first non-null value.

Every `<resource>Api.ts` imports `buildQueryString` from `utils`; do **not** redefine it locally.

## Key conventions

- **Keep the mock path and the commented real path in sync.** When adding or renaming a query param, update both the commented `url`/`body` line and the `MockXxx` fixture shape — otherwise the package silently diverges from production.
- **Use underscore suffixes in TS, rely on `translateParamName` for the wire.** `QueryParams` interfaces and `getX` parameters use `entityRegistryIds_anyOf`, `emptyPositions_gt`, etc. Never URL-encode these raw — always go through `buildQueryString`, which applies the dot translation.
- **JSON-shaped inputs are passed as strings, parsed inside `createX`.** Array/object fields (`aliases`, `annotations`, `fields`, `customFields`, `wells`, `ingredients`, `fieldValidation`, …) come in as JSON strings from the `InputForm` and are `JSON.parse`d only when non-empty. Don't change this to typed arrays without also teaching the standard function preview how to render them.
- **All list responses go through `dataFrameFromObjects`.** New resource files should delegate to it rather than hand-building columns.
- **Date range filters come in quartets of `_gt` / `_gte` / `_lt` / `_lte`** (see `getPlates`, `getAssayResults`) — translated to the swagger's `.gt`/`.gte`/`.lt`/`.lte` automatically.
- **Tree order is intentional, not alphabetical** (AA Sequences → DNA Sequences → Assay Results → Assay Runs → Molecules → Plates → Mixtures → DNA Oligos → Projects). Preserve this unless explicitly asked to resort.
- **Use `grok.dapi.fetchProxy`, never raw `fetch`**, for Benchling calls — `benchling.com` is an external origin and CORS will block a direct `fetch`.
- **Do not construct custom views in the tree browser.** Use `DG.ObjectHandler.forEntity(func).renderPreview(func)` + `grok.shell.preview = …`. If you feel tempted to wire `InputForm`, `splitV`, `userSettings` persistence, or ribbon buttons — stop; the default platform preview already does it.

## Schema-matched create shapes (must match `swaggers/benchling_openapi.yaml`)

The POST bodies are specific, and several create endpoints are **bulk-shaped** even for a single item:

- `POST /aa-sequences` — `AaSequenceCreate`; required `name`, `aminoAcids`.
- `POST /dna-sequences` — `DnaSequenceCreate`; required `name`, `bases`, **`isCircular`**.
- `POST /dna-oligos` — `DnaOligoCreate`; required `name`, `bases`. Optional `annotations`, `helm`, custom notation fields.
- `POST /molecules` — `MoleculeCreate`; required `name` and **`chemicalStructure: { structureFormat, value }`** (NOT flat `smiles`/`formula`).
- `POST /mixtures` — `MixtureCreate`; required `name`, `ingredients`, `schemaId`, `units`.
- `POST /plates` — `PlateCreate`; required `schemaId`.
- `POST /assay-results` — **bulk envelope** `AssayResultsBulkCreateRequest` = `{ assayResults: [AssayResultCreate, ...] }`; each item requires `schemaId`, `fields`.
- `POST /assay-runs` — **bulk envelope** `AssayRunsBulkCreateRequest` = `{ assayRuns: [AssayRunCreate, ...] }`; each item requires `schemaId`, `fields`.

For the two bulk endpoints, the `createAssayResult` / `createAssayRun` wrappers in `package.ts` accept a single item's worth of inputs and wrap it into the `{ assayResults: [...] }` / `{ assayRuns: [...] }` envelope before posting. The poster functions are named **`postAssayResults` / `postAssayRuns`** (plural) and return `{ assayResults: string[] }` / `{ assayRuns: string[] }` — IDs, not full entities.

The `Molecule` response shape has **no `smiles` field** — only `canonicalizedSmiles` and `originalSmiles`. The TS `Molecule` interface reflects this; do not add a `smiles` property.

`GET /assay-runs` requires `schemaId` (per swagger); `AssayRunsQueryParams.schemaId` and `getAssayRuns(schemaId, …)` are both non-optional.

## Adding a new Benchling resource

1. Create `src/<resource>Api.ts` following the 7-step template above. Copy from `dnaSequencesApi.ts` (simpler) or `platesApi.ts` (has `_gt`/`_lte` quartets). Import `buildQueryString` and `dataFrameFromObjects` from `./utils`.
2. Match every request body to the swagger's `*Create` schema exactly — check for required fields (like `isCircular` on DNA) and bulk envelopes (like `{ assayResults: [...] }` on assay POSTs) before writing TS.
3. Add `queryXxx`, `postXxx` imports to `src/package.ts`.
4. Add `getXxx` and `createXxx` static methods on `PackageFunctions`, with a `@grok.decorators.param({options: {nullable: true}})` for every optional filter.
5. In `benchlingLinkAppTreeBrowser`, add one line: `addFuncNode('<Display>', 'BenchlingLink:getXxx');`.
6. `npm run build` so `grok api` regenerates `package.g.ts` / `package-api.ts`.

## Activating real HTTP for a resource

1. In the target `<resource>Api.ts`, uncomment the `// const token = ... / url / response / data / df` block in `queryXxx` and `postXxx`.
2. Replace `'YOUR_BENCHLING_API_TOKEN'` with `await getApiKey()` and add `import { getApiKey } from './credentials';`.
3. The commented block already uses `buildQueryString(params)` — since that now lives in `utils.ts` and applies dot-translation, the `_anyOf`/`_gt`/etc suffixes will be correctly rewritten on the wire.
4. Delete (or gate behind a `mock` flag) the `return mockXxx...` line.
5. Configure the `apiKey` credential on the deployed package (`grok s` → package credentials, parameter name `apiKey`).

## Benchling glossary (for naive devs)

- **AA Sequence** — amino-acid sequence (protein). Field: `aminoAcids`.
- **DNA Sequence** — full DNA construct. Fields: `bases`, required `isCircular` on create.
- **DNA Oligo** — short synthetic DNA. Same `bases` field plus `helm` / `customNotationId`. Distinct endpoint from DNA Sequence.
- **Molecule** — small chemical. Response: `canonicalizedSmiles`, `originalSmiles` (NO flat `smiles`). Create: `chemicalStructure: { structureFormat: 'smiles' | 'molfile', value }`.
- **Mixture** — composed entity; `ingredients` (array) is required along with `schemaId` and `units`.
- **Plate** — physical container with `wells`, `barcode`; plate *filter* params use `emptyPositions`/`emptyContainers` counters (plus `_gt`/`_gte`/`_lt`/`_lte`), but those counters are not fields on the `Plate` entity itself.
- **Assay Result** vs **Assay Run** — a run groups many results; both are schema-driven and typically filtered by `schemaId`. Both POSTs take a bulk envelope (`{assayResults: [...]}` / `{assayRuns: [...]}`) and return ID arrays, not full entities. `GET /assay-runs` *requires* `schemaId`.
- **Project** — top-level organizational unit. No `POST /projects` in the API, so no `createProject`.
- **Schema** (`schemaId`) — user-defined field template on the Benchling tenant; drives which `fields` a create call accepts.
- **Registry** (`registryId`) — the tenant's controlled registry of named entities.
- **Folder** (`folderId`) — workspace folder (distinct from Registry).
- **`_anyOf` / `_anyOf_caseSensitive` (TS side) ↔ `.anyOf` / `.anyOf.caseSensitive` (wire side)** — Benchling convention for multi-value OR filters. `utils.translateParamName` handles the dot conversion.
