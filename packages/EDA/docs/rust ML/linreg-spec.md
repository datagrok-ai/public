# Спецификация: линейная регрессия с Elastic Net на Rust + WASM

**Версия:** 1.1
**Цель:** реализовать с нуля (на голом `Vec<f64>`) обучаемую градиентным спуском линейную регрессию с одновременной L1- и L2-регуляризацией (Elastic Net). Rust-ядро универсально; WASM — один из бэкендов.

> **Изменение в v1.1.** Данные хранятся **по-колоночно** и подаются как **список колонок** (`&[&[f64]]`). Масштабирование выполняется на стороне **JS/TS (вариант B)**: в ядро приходят уже стандартизованные колонки, а μ/σ передаются отдельно и используются **только** для раскрутки коэффициентов в исходное пространство. См. разделы 3.1, 4.2, 4.4, 5 и 6.

---

## 1. Область применения и ограничения

В объёме (in scope):
- Мультипризнаковая линейная регрессия `ŷ = Xw + b`.
- Обучение мини-батчевым / полнобатчевым градиентным спуском.
- Elastic Net: два независимых коэффициента `λ₁` (L1) и `λ₂` (L2), действующие одновременно.
- Корректная обработка недифференцируемости L1 через proximal-шаг (soft-thresholding, ISTA).
- Масштабирование признаков выполняется вне ядра (на JS/TS); ядро принимает стандартизованные данные, μ/σ — только для раскрутки коэффициентов.
- Метрики: MSE, RMSE, MAE, R².
- WASM-обёртка через `wasm-bindgen`.

Вне объёма (out of scope) в v1.0:
- Аналитическое нормальное уравнение (только GD).
- Полиномиальные/взаимодействующие признаки (пользователь готовит их сам).
- Многомерный таргет (только скалярный `y`).
- Многопоточность / SIMD (возможное расширение).

## 2. Математическая модель

### 2.1. Предсказание

```
ŷᵢ = Σⱼ wⱼ·xᵢⱼ + b
```

### 2.2. Целевая функция

Усреднённый MSE плюс два штрафа. Смещение `b` **не регуляризуется**.

```
L(w, b) = (1/n)·Σᵢ (ŷᵢ − yᵢ)²  +  λ₁·‖w‖₁  +  (λ₂/2)·‖w‖₂²
```

где `‖w‖₁ = Σⱼ|wⱼ|`, `‖w‖₂² = Σⱼ wⱼ²`.

> **Соглашение о масштабе.** MSE усредняется по `n`. Тогда `λ₁`, `λ₂` имеют один и тот же масштаб независимо от размера выборки — это важно для воспроизводимости при смене `n` и батч-сайза. Градиент гладкой части также делится на `n`.

### 2.3. Градиент гладкой части

Гладкая часть `f(w,b) = MSE + (λ₂/2)·‖w‖₂²` дифференцируема:

```
r = ŷ − y                         (вектор остатков, длина n)

∂f/∂wⱼ = (2/n)·Σᵢ rᵢ·xᵢⱼ  +  λ₂·wⱼ
∂f/∂b  = (2/n)·Σᵢ rᵢ
```

L2-штраф входит в гладкий градиент. L1-штраф **не** дифференцируем в нуле, поэтому в градиент не включается — он обрабатывается отдельным proximal-шагом.

> **Column-major.** Сумма `Σᵢ rᵢ·xᵢⱼ` — это скалярное произведение вектора остатков `r` на колонку `j`. При колоночном хранении колонка `X[:,j]` лежит непрерывно, поэтому весь градиент `∇w` вычисляется как `n_cols` независимых dot-product'ов, каждый — линейный проход по непрерывной памяти. Остатки `r` считаются один раз на батч и переиспользуются для всех столбцов.

### 2.4. Proximal gradient (ISTA) для L1

Один шаг обучения:

1. **Градиентный шаг по гладкой части:**
   ```
   w_tmp = w − lr·∇w f
   b     = b − lr·∂f/∂b        (b обновляется обычным градиентом, без prox)
   ```
2. **Proximal-шаг (soft-thresholding) по L1:**
   ```
   wⱼ = soft(w_tmpⱼ, lr·λ₁)

   soft(z, τ) = sign(z)·max(|z| − τ, 0)
   ```

Soft-thresholding честно зануляет малые веса (свойство Lasso) — это причина, по которой простой субградиент L1 нежелателен: он «дрожит» вокруг нуля и не даёт точных нулей.

> Если `λ₁ = 0`, prox-шаг вырождается в тождество → получаем чистый Ridge.
> Если `λ₂ = 0`, гладкий градиент теряет L2-член → чистый Lasso.
> Если оба нуля → обычная МНК-регрессия через GD.

## 3. Предобработка данных

### 3.1. Масштабирование (выполняется на стороне JS/TS — вариант B)

Без масштабирования штрафы несопоставимы между признаками разного масштаба. Стандартизация по столбцам:

```
X_scaledᵢⱼ = (xᵢⱼ − μⱼ) / σⱼ      (если σⱼ ≈ 0 → σⱼ = 1, столбец-константа)
```

**В v1.1 (вариант B) масштабирование выполняет вызывающий код (JS/TS).** В ядро подаются **уже стандартизованные** колонки, а μ и σ для каждой колонки передаются отдельно — но не для трансформации данных (она уже сделана), а **только** для раскрутки обученных коэффициентов обратно в исходное пространство признаков (см. 4.4).

Из этого следует:
- Ядро **не** применяет `(x−μ)/σ` к входным данным — `transform` отсутствует/тождественен. Колонки в `fit` и `predict` обязаны быть в одном и том же (стандартизованном) масштабе; ответственность за это на вызывающей стороне.
- Внутренний градиентный спуск работает прямо над поданными (стандартизованными) колонками.
- μ/σ хранятся в модели исключительно ради `coefficients()`. Если они не заданы, обучение и `predict` корректны, но `coefficients()` сможет вернуть веса только в стандартизованном пространстве (см. 4.4).

Контракт статистик (когда заданы):
- `means.len() == stds.len() == n_cols`;
- все значения конечны (нет NaN/Inf) → иначе `NonFiniteValue`;
- `σⱼ > 0`; `σⱼ` ниже порога `eps` (например `1e-12`) трактуется как константный столбец (`σ = 1`); отрицательное `σⱼ` → `InvalidHyperparameter`.

> **Важно про утечку данных.** Поскольку масштабирование вне ядра, корректность μ/σ — ответственность JS/TS-пайплайна: статистики обязаны считаться **только на train** и теми же значениями применяться к valid/test/inference. Ядро не может это проконтролировать.

> **Внутренняя согласованность `predict`.** Так как ядро не масштабирует, на `predict` должны подаваться колонки в том же стандартизованном масштабе, что и на `fit`. Это инвариант контракта; ядро не имеет способа его проверить и полагается на вызывающую сторону.

### 3.2. Таргет

`y` не масштабируется (упрощает интерпретацию `b` и метрик в исходных единицах). Опционально можно центрировать `y` — отложено.

## 4. Архитектура Rust-ядра

Принцип: ядро не знает ни про WASM, ни про JS. Чистый `#![no_std]`-совместимый по духу код на `Vec<f64>` (хотя `std` оставляем ради `Vec` и `f64::sqrt`).

### 4.1. Структура крейта

```
crates/
  linreg-core/          # чистое ядро, без wasm
    src/
      lib.rs
      matrix.rs         # минимальные операции над Vec<f64> (column-major)
      scaler.rs         # FeatureStats: хранение μ/σ и раскрутка коэффициентов
      model.rs          # ElasticNet: fit / predict
      metrics.rs        # mse, rmse, mae, r2
      error.rs          # типы ошибок
    tests/
      integration.rs    # см. раздел 7
  linreg-wasm/          # обёртка wasm-bindgen, зависит от linreg-core
    src/lib.rs
```

### 4.2. Представление данных — список колонок (column-major)

**Входной контракт (фиксирован).** На вход подаётся **список колонок**: `columns: &[&[f64]]`, где `columns[j]` — весь признак `j`, непрерывный массив длиной `n_rows`. Число признаков `n_cols = columns.len()`; `n_rows` выводится из длины первой колонки. Это основной и единственный публичный способ подачи `X` — отдельный параметр `n_cols`/`n_rows` для матрицы не нужен.

Инварианты, проверяемые при входе:
- `columns` не пуст → иначе `EmptyDataset`;
- все колонки одной длины → иначе `DimensionMismatch`;
- `y.len() == n_rows` → иначе `DimensionMismatch`;
- ни одна колонка не пуста (`n_rows > 0`) → иначе `EmptyDataset`.

**Внутреннее представление.** Ядро может либо работать прямо по `&[&[f64]]`, либо один раз сконкатенировать колонки в плоский column-major буфер `Vec<f64>` (признак `j` → `[j*n_rows, (j+1)*n_rows)`) для локальности и единообразия. Выбор — деталь реализации, не видимая в API.

Почему это удобно: штрафы L1/L2 и градиент по `wⱼ` — операции «над целой колонкой». Колонка уже непрерывна → каждый проход кэш-дружелюбен. Транспонирование не требуется никогда: данные приходят ровно в той форме, в какой их потребляет горячий цикл. (Масштабирование к ядру не относится — оно сделано на стороне JS/TS, вариант B.)

**Цена:** предсказание `ŷᵢ = Σⱼ wⱼ·xᵢⱼ` логически идёт по строкам. Решается аккумуляторной схемой по столбцам (раздел 5): `ŷ` инициализируется `b`, затем для каждой колонки `j` выполняется `ŷ += wⱼ · columns[j]`. Память по-прежнему читается колонками.

### 4.3. Ключевые типы (эскиз API)

```rust
// error.rs
#[derive(Debug, Clone, PartialEq)]
pub enum LinRegError {
    DimensionMismatch { expected: usize, got: usize },
    EmptyDataset,
    NotFitted,
    InvalidHyperparameter(String),
    NonFiniteValue,        // NaN/Inf в данных или при расходимости
}

// model.rs
pub struct Config {
    pub lr: f64,            // learning rate,   default 0.01
    pub epochs: usize,      // default 1000
    pub l1: f64,            // λ₁,  default 0.0
    pub l2: f64,            // λ₂,  default 0.0
    pub batch_size: usize,  // 0 == full-batch (default)
    pub tol: f64,           // ранняя остановка по Δloss, default 1e-6
    pub seed: u64,          // для перемешивания мини-батчей, default 42
    pub fit_intercept: bool,// default true
    // NB: флага standardize нет — масштабирование выполняет вызывающий код (вариант B).
    // Ядро всегда обучается над поданными (уже стандартизованными) колонками.
}

impl Default for Config { /* значения выше */ }

// scaler.rs — в варианте B хранит μ/σ ТОЛЬКО для раскрутки коэффициентов.
// Никакого transform над данными ядро не выполняет.
pub struct FeatureStats { means: Vec<f64>, stds: Vec<f64> } // σ≈0 нормализованы к 1.0

impl FeatureStats {
    pub fn new(means: Vec<f64>, stds: Vec<f64>) -> Result<Self, LinRegError>; // валидирует
    pub fn means(&self) -> &[f64];
    pub fn stds(&self)  -> &[f64];
}

pub struct ElasticNet {
    weights: Vec<f64>,           // в стандартизованном пространстве (как поданы данные)
    bias: f64,
    stats: Option<FeatureStats>, // только для раскрутки coefficients()
    config: Config,
    history: Vec<f64>,           // loss по эпохам (для диагностики/тестов)
}

impl ElasticNet {
    pub fn new(config: Config) -> Result<Self, LinRegError>; // валидирует гиперпараметры

    // Опционально: задать μ/σ для раскрутки коэффициентов в исходное пространство.
    // НЕ влияет на обучение/предсказание (данные уже стандартизованы вызывающим кодом).
    // Длины должны равняться n_cols. Без вызова coefficients() вернёт веса
    // в стандартизованном пространстве.
    pub fn set_feature_stats(&mut self, means: Vec<f64>, stds: Vec<f64>)
        -> Result<(), LinRegError>;

    // КОНТРАКТ: columns — УЖЕ СТАНДАРТИЗОВАННЫЕ колонки.
    //   columns[j] — признак j, длиной n_rows; n_cols = columns.len().
    pub fn fit(&mut self, columns: &[&[f64]], y: &[f64]) -> Result<(), LinRegError>;
    // predict: columns должны быть в ТОМ ЖЕ масштабе, что и на fit. μ/σ не применяются.
    pub fn predict(&self, columns: &[&[f64]]) -> Result<Vec<f64>, LinRegError>;

    pub fn loss_history(&self) -> &[f64];
    // Если μ/σ заданы → веса в ИСХОДНОМ пространстве; иначе → в стандартизованном.
    pub fn coefficients(&self) -> (Vec<f64>, f64);
}
```

> В варианте B `set_feature_stats` опционален и влияет **только** на `coefficients()`. Обучение и `predict` работают над поданными (стандартизованными) колонками и μ/σ не используют. Ответственность за единый масштаб данных между `fit` и `predict` — на вызывающей стороне.

### 4.4. Раскрутка масштабирования в коэффициенты

Обучение идёт над стандартизованными колонками, поэтому веса `w` и смещение `b` живут в стандартизованном пространстве. Чтобы `coefficients()` вернул интерпретируемые веса для **исходных** (немасштабированных) данных, μ/σ применяются к уже обученным параметрам:

```
w_origⱼ = wⱼ / σⱼ
b_orig  = b − Σⱼ wⱼ·μⱼ / σⱼ
```

Это единственное место, где ядру нужны μ/σ. Поведение `coefficients()`:
- μ/σ заданы (`set_feature_stats`) → возвращаются веса в исходном пространстве по формулам выше;
- μ/σ не заданы → возвращаются веса в стандартизованном пространстве (как есть), что отмечается в документации метода; формула раскрутки недоступна.

> `predict`, напротив, μ/σ не использует никогда: он работает в том же стандартизованном пространстве, в котором поданы данные (вариант B). Раскрутка нужна исключительно для интерпретации коэффициентов человеком.

## 5. Алгоритм обучения (псевдокод)

```
fit(columns, y):                            # columns: УЖЕ СТАНДАРТИЗОВАННЫЕ колонки
    n_cols = columns.len();  n_rows = columns[0].len()
    валидировать инварианты входа (раздел 4.2) → EmptyDataset / DimensionMismatch
    проверить отсутствие NaN/Inf → NonFiniteValue
    # масштабирования НЕТ: данные пришли стандартизованными (вариант B).
    # μ/σ, если заданы через set_feature_stats, не трогаются здесь — только для coefficients().
    w = zeros(n_cols); b = 0
    prev_loss = +inf
    for epoch in 0..epochs:
        для каждого батча B (набор индексов строк):
            # --- предсказание: аккумулятор по столбцам, не по строкам ---
            ŷ = filled(|B|, b)
            for j in 0..n_cols:
                col = columns[j]               # непрерывный массив признака j
                for k, i in enumerate(B): ŷ[k] += w[j] · col[i]
            r = ŷ − y_B
            # --- градиент: по одному dot-product на колонку ---
            grad_b = (2/|B|)·Σ r
            for j in 0..n_cols:
                col = columns[j]
                g = 0
                for k, i in enumerate(B): g += r[k] · col[i]
                grad_w[j] = (2/|B|)·g + l2·w[j]
            # --- шаг + prox ---
            w = w − lr·grad_w
            if fit_intercept: b = b − lr·grad_b
            w = soft_threshold(w, lr·l1)
        loss = compute_loss(columns, y, w, b)   # на всей выборке, та же аккумуляторная схема
        history.push(loss)
        if !loss.is_finite(): return Err(NonFiniteValue)
        if |prev_loss − loss| < tol: break
        prev_loss = loss
```

> Оба горячих прохода (предсказание и градиент) организованы как внешний цикл по колонкам и внутренний по строкам батча — это сохраняет последовательный доступ к памяти. Для full-batch (`batch_size=0`) `B` — все строки, и `col[i]` превращается в простой последовательный обход колонки.

## 6. WASM-бэкенд

Тонкая обёртка. Ядро остаётся чистым.

```rust
// linreg-wasm/src/lib.rs
use wasm_bindgen::prelude::*;
use linreg_core::{ElasticNet, Config};

#[wasm_bindgen]
pub struct WasmElasticNet { inner: ElasticNet }

#[wasm_bindgen]
impl WasmElasticNet {
    #[wasm_bindgen(constructor)]
    pub fn new(lr: f64, epochs: usize, l1: f64, l2: f64) -> Result<WasmElasticNet, JsError> {
        let cfg = Config { lr, epochs, l1, l2, ..Default::default() };
        Ok(Self { inner: ElasticNet::new(cfg).map_err(to_js)? })
    }

    // μ/σ из JS/TS — по одному значению на колонку (Float64Array).
    // В варианте B нужны ТОЛЬКО для раскрутки coefficients() в исходное пространство;
    // на fit/predict не влияют (данные уже стандартизованы на стороне JS). Опционально.
    pub fn set_feature_stats(&mut self, means: &[f64], stds: &[f64])
        -> Result<(), JsError> {
        self.inner.set_feature_stats(means.to_vec(), stds.to_vec()).map_err(to_js)
    }

    // wasm-bindgen не передаёт &[&[f64]] напрямую. Список колонок приходит
    // как один плоский Float64Array (конкатенация колонок) + n_rows.
    // На границе восстанавливаем срезы колонок и зовём ядро.
    pub fn fit(&mut self, flat_columns: &[f64], n_rows: usize, y: &[f64])
        -> Result<(), JsError> {
        let cols = to_columns(flat_columns, n_rows).map_err(to_js)?;
        self.inner.fit(&cols, y).map_err(to_js)
    }

    pub fn predict(&self, flat_columns: &[f64], n_rows: usize)
        -> Result<Vec<f64>, JsError> {
        let cols = to_columns(flat_columns, n_rows).map_err(to_js)?;
        self.inner.predict(&cols).map_err(to_js)
    }
}

// flat_columns.len() должно делиться на n_rows; n_cols выводится делением.
fn to_columns(flat: &[f64], n_rows: usize) -> Result<Vec<&[f64]>, LinRegError> {
    if n_rows == 0 || flat.len() % n_rows != 0 { /* -> DimensionMismatch */ }
    Ok(flat.chunks_exact(n_rows).collect())
}
```

Сборка: `wasm-pack build crates/linreg-wasm --target web`.

Замечания по границе JS↔WASM. Ядро принимает список колонок (`&[&[f64]]`), но `wasm-bindgen` не умеет передавать массив массивов как непрерывную типизированную структуру. Поэтому контракт границы такой: JS конкатенирует колонки в **один плоский `Float64Array`** (порядок: колонка за колонкой, то есть `[col0 целиком, col1 целиком, …]`) и передаёт его вместе с `n_rows`. На стороне Rust `chunks_exact(n_rows)` бесплатно (без копий) восстанавливает срезы колонок поверх линейной памяти WASM. Поскольку JS уже хранит данные по колонкам, конкатенация — единственная подготовительная операция, транспонирование не нужно. `Vec<f64>` на выходе `predict` материализуется в `Float64Array` автоматически. Альтернатива (массив отдельных `Float64Array`, по одному на колонку) возможна, но даёт N мелких копий вместо одной — оставлена как опция, не дефолт.

## 7. Валидационные тесты

Тесты делятся на: (а) unit-тесты ядра, (б) числовые/property-тесты на корректность алгоритма, (в) тесты WASM-границы. Эталоны вычисляются аналитически или сверяются с известными свойствами — без внешних ML-зависимостей.

### 7.1. Unit-тесты компонентов

| # | Что проверяем | Ожидание |
|---|---|---|
| U1 | `FeatureStats::new` с валидными μ/σ | конструируется; `means()`/`stds()` возвращают поданное (σ≈0 → 1.0) |
| U2 | `FeatureStats::new` с `σⱼ ≈ 0` (ниже eps) | σ нормализуется к 1.0, без паники |
| U2d | `FeatureStats::new` с отрицательным `σⱼ` или NaN/Inf в μ/σ | `Err(InvalidHyperparameter)` / `Err(NonFiniteValue)` |
| U2e | `set_feature_stats` с длиной ≠ n_cols (выясняется на fit/coefficients) | `Err(DimensionMismatch)` |
| U2g | **`fit`/`predict` игнорируют μ/σ**: обучить с заданными и с незаданными μ/σ на одних и тех же (стандартизованных) данных | веса модели и `predict` идентичны (μ/σ на них не влияют) |
| U2h | **Раскрутка vs стандартизованные веса**: `coefficients()` с заданными μ/σ против без них | без μ/σ — веса в стандартизованном пространстве; с μ/σ — пересчёт по формулам 4.4 |
| U3 | `soft_threshold(z, τ)` | `soft(5,2)=3`, `soft(-5,2)=-3`, `soft(1,2)=0` |
| U4 | Метрики на известных векторах | MSE/RMSE/MAE/R² совпадают с ручным расчётом |
| U5 | `R²` для идеального предсказания | `R² = 1.0` (в пределах 1e-9) |
| U6 | Валидация: отрицательный `lr`, `epochs=0`, `l1<0` | `Err(InvalidHyperparameter)` |
| U7 | NaN/Inf во входных `X` или `y` | `Err(NonFiniteValue)` |
| U8 | Несовпадение размеров `len(X) ≠ n_rows*n_cols` | `Err(DimensionMismatch)` |
| U9 | `predict` до `fit` | `Err(NotFitted)` |
| U10 | **Контракт колонок**: `fit` принимает `&[&[f64]]`, признак `j` = `columns[j]` | веса соответствуют поданным колонкам |
| U11 | **Колонки разной длины** (`columns[0].len() != columns[1].len()`) | `Err(DimensionMismatch)` |
| U12 | **Пустой список колонок** или пустая колонка | `Err(EmptyDataset)` |
| U13 | **WASM-граница**: плоская конкатенация + `n_rows`, восстановление `chunks_exact(n_rows)` | совпадает с прямым вызовом ядра на тех же колонках |
| U14 | **Неделимость на WASM-границе** (`flat.len() % n_rows != 0`) | `Err(DimensionMismatch)` |

### 7.2. Корректность обучения (числовые тесты)

| # | Сценарий | Ожидание |
|---|---|---|
| C1 | **Идеальная линия без шума**, `y = 3x + 2`, `λ₁=λ₂=0` | `w≈3`, `b≈2` (tol 1e-3); итоговый MSE ≈ 0 |
| C2 | **Многопризнаковая** `y = 1·x₁ − 2·x₂ + 0.5·x₃ + 4` без шума, на стандартизованном входе + μ/σ | `coefficients()` (с раскруткой) ≈ истинным в исходном пространстве (tol 1e-2) |
| C3 | **Монотонное убывание loss** при `λ₁=λ₂=0` и адекватном `lr` | `history` не возрастает (с допуском на мини-батч-шум) |
| C4 | **Ridge сжимает веса**: один и тот же датасет, `λ₂=0` vs `λ₂=10` | `‖w‖₂` при `λ₂=10` строго меньше |
| C5 | **Lasso зануляет**: добавить «мусорный» некоррелированный признак, `λ₁` большой | вес мусорного признака точно `0.0` |
| C6 | **Elastic Net = оба эффекта**: коррелированная группа + шум, `λ₁,λ₂>0` | часть весов занулена И общая норма сжата |
| C7 | **Раскрутка корректна**: обучить на стандартизованных данных, задать μ/σ | `coefficients()` в исходном пространстве совпадает с эталоном `wⱼ/σⱼ`, `b−Σwⱼμⱼ/σⱼ` (tol 1e-9) |
| C8 | **Согласованность predict/coefficients**: предсказание исходных `y` = (раскрученные коэф.)·(исходные X) | совпадает с `predict()` на стандартизованных X (tol 1e-9) |
| C9 | **Расходимость при большом `lr`** | возвращается `Err(NonFiniteValue)`, не паника |
| C10 | **Ранняя остановка**: `tol` достигнут | число эпох в `history` < `epochs` |
| C11 | **Детерминизм**: два `fit` с одним `seed` | идентичные веса побайтово |
| C12 | **Эквивалентность full-batch vs batch_size=n** | совпадающие результаты |

### 7.3. Property-based тесты (`proptest`)

| # | Свойство |
|---|---|
| P1 | Для случайных линейных данных без шума финальный train-MSE < ε при λ=0 |
| P2 | Увеличение `λ₁` монотонно не увеличивает `‖w‖₁` (на сетке значений) |
| P3 | `predict` не возвращает NaN/Inf на любых конечных входах после успешного `fit` |
| P4 | Перестановка строк `(X, y)` при full-batch не меняет финальные веса (tol) |

### 7.4. Тесты WASM-границы

Запуск через `wasm-bindgen-test` в headless-браузере/Node.

| # | Что проверяем |
|---|---|
| W1 | Плоская конкатенация колонок (`Float64Array`) + `n_rows` корректно читается; `fit` отрабатывает без копий-ошибок |
| W2 | `predict` возвращает `Float64Array` правильной длины |
| W3 | Ошибки ядра (`DimensionMismatch` и т.п.) прокидываются как `JsError`/исключение JS |
| W4 | Числовой паритет: один датасет через нативный `cargo test` и через WASM дают совпадающие веса (tol 1e-9) |
| W5 | `set_feature_stats(means, stds)` из JS → влияет только на `coefficients()` (раскрутка), не на predict |

### 7.5. Эталонная сверка (опционально, в CI отдельной задачей)

Сгенерировать небольшой датасет, прогнать офлайн в `scikit-learn` (`ElasticNet`/`SGDRegressor`), сохранить ожидаемые коэффициенты в JSON-фикстуру и сверять с допуском. Это не зависимость кода, а только артефакт теста — допустимо при политике «ядро без внешних крейтов».

## 8. Гиперпараметры по умолчанию (сводка)

| Параметр | Default | Обоснование |
|---|---|---|
| `lr` | `0.01` | безопасно для стандартизованных данных |
| `epochs` | `1000` | с ранней остановкой по `tol` |
| `l1` (λ₁) | `0.0` | по умолчанию регуляризация выключена |
| `l2` (λ₂) | `0.0` | то же |
| `batch_size` | `0` (full-batch) | детерминизм и простота |
| `tol` | `1e-6` | ранняя остановка |
| `seed` | `42` | воспроизводимость |
| `fit_intercept` | `true` | `b` не регуляризуется |

Масштабирования в ядре нет: данные подаются уже стандартизованными (вариант B). μ/σ задаются опционально через `set_feature_stats` и используются только для раскрутки `coefficients()` в исходное пространство.

## 9. Этапы разработки

1. **Каркас** — два крейта, `Config::default`, типы ошибок, заглушки API. Тесты U6–U9.
2. **FeatureStats + раскрутка** — `FeatureStats::new(means, stds)` с валидацией (длины, конечность, σ>0, σ≈0→1.0); хранение μ/σ в модели; раскрутка `coefficients()` по формулам 4.4. Ядро НЕ масштабирует данные. Тесты U1–U2, U2d–U2h, U10–U12.
3. **Метрики** — MSE/RMSE/MAE/R². Тесты U4–U5.
4. **Гладкий GD (без штрафов)** — `fit`/`predict`, история loss. Тесты C1–C3, C12.
5. **L2 (Ridge)** — добавить `λ₂·w` в градиент. Тест C4.
6. **L1 через soft-thresholding (ISTA)** — prox-шаг. Тесты U3, C5.
7. **Elastic Net** — оба штрафа вместе. Тест C6.
8. **Раскрутка коэффициентов + инвариантность масштаба.** Тесты C7–C8.
9. **Устойчивость** — детекция расходимости, ранняя остановка, детерминизм. Тесты C9–C11.
10. **Property-тесты.** P1–P4.
11. **WASM-обёртка** + `wasm-pack`: плоская конкатенация колонок на границе, реконструкция через `chunks_exact`. Тесты W1–W4, U13–U14.
12. **CI** — `cargo test`, `wasm-pack test`, опциональная sklearn-фикстура (7.5).
