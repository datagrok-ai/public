# Спецификация: PCA через NIPALS на Rust + WASM

**Версия:** 1.0
**Цель:** реализовать с нуля (на голом `Vec<f64>`) метод главных компонент (PCA) алгоритмом NIPALS — итеративным извлечением компонент по одной с дефляцией, без явного формирования ковариационной матрицы. Rust-ядро универсально; WASM — один из бэкендов.

> **Общий контракт с `linreg-spec` (вариант B).** Данные подаются как **список колонок** (`&[&[f64]]`, по одному непрерывному массиву на признак). Центрирование и (при необходимости) масштабирование выполняются на стороне **JS/TS**: в ядро приходят уже подготовленные колонки. μ/σ передаются **опционально** и используются только для раскрутки нагрузок/проекций в исходное пространство признаков, а не для трансформации данных внутри ядра. Та же система ошибок и та же WASM-граница (плоская конкатенация колонок + `n_rows`).

---

## 1. Область применения и ограничения

В объёме (in scope):
- Извлечение `k` главных компонент алгоритмом NIPALS.
- Нагрузки `P` (loadings), scores `T` обучающих данных, объяснённая дисперсия и её доля по компонентам, сингулярные значения.
- Проекция новых данных `transform(columns)` и обратная реконструкция `inverse_transform`.
- Опциональная реортогонализация (Gram–Schmidt) для борьбы с накоплением ошибки дефляции.
- Детерминированное соглашение о знаке компонент (для воспроизводимости и тестов).
- WASM-обёртка через `wasm-bindgen`.

Вне объёма (out of scope) в v1.0:
- Центрирование/масштабирование внутри ядра (делается на JS, вариант B).
- Ядерный (kernel) PCA, разреженный PCA, рандомизированный SVD.
- Автоматический выбор числа компонент (Кайзер, локоть, кросс-валидация) — `k` задаётся пользователем.
- Обработка пропусков (NIPALS это умеет, но в v1.0 вход обязан быть полным).

## 2. Математическая модель

### 2.1. Постановка

Дана матрица `X` размера `n × m` (`n` строк-наблюдений, `m` признаков-колонок), **уже центрированная по столбцам** (среднее каждого столбца ≈ 0; центрирование сделано на стороне JS). PCA ищет разложение:

```
X ≈ T · Pᵀ = Σₐ tₐ · pₐᵀ        (a = 1..k)
```

где `pₐ` — нагрузки `a`-й компоненты (единичный вектор в пространстве признаков, длина `m`), `tₐ` — scores (проекции наблюдений на компоненту, длина `n`). Нагрузки `pₐ` — это собственные векторы выборочной ковариационной матрицы `XᵀX/(n−1)`, а дисперсия `tₐ` — соответствующее собственное значение.

### 2.2. Объяснённая дисперсия

```
варₐ            = (tₐᵀ tₐ) / (n − 1)            # дисперсия a-й компоненты (собственное значение)
сингулярное σₐ  = ‖tₐ‖ = sqrt(tₐᵀ tₐ)           # сингулярное значение X
доля_варₐ       = варₐ / total_var
total_var       = Σⱼ Var(X[:,j])                # сумма дисперсий центрированных колонок = trace(cov)
```

`total_var` вычисляется один раз по исходным (поданным) центрированным колонкам до дефляции.

## 3. Алгоритм NIPALS

### 3.1. Извлечение одной компоненты

Для текущей (дефлированной) матрицы `X`:

```
выбрать стартовый score t ← столбец X с максимальной дисперсией
повторять до сходимости или max_iter:
    p = Xᵀ t / (tᵀ t)          # проекция на пространство признаков (нагрузки)
    p = p / ‖p‖                # нормировка нагрузок к единичной длине
    t_new = X p                # обновление scores (‖p‖=1 ⇒ делить на pᵀp не нужно)
    если ‖t_new − t‖ / ‖t_new‖ < tol: t = t_new; стоп
    t = t_new
вернуть (t, p)
```

### 3.2. Дефляция и цикл по компонентам

```
total_var = Σⱼ Var(X[:,j])
X_work = копия X                      # ядро не мутирует вход вызывающего
для a в 1..=k:
    если все остаточные дисперсии ≈ 0: стоп (ранг исчерпан) → вернуть < k компонент
    (t, p) = extract_component(X_work)
    [опц.] реортогонализовать t против {t₁..t_{a-1}}, p против {p₁..p_{a-1}} (Gram–Schmidt)
    зафиксировать знак(p) детерминированно (см. 3.4)
    варₐ = tᵀt/(n−1); σₐ = ‖t‖
    сохранить pₐ (строка P), tₐ (столбец T)
    X_work = X_work − t·pᵀ            # дефляция: убрать вклад компоненты
```

### 3.3. Численная устойчивость

- **Накопление ошибки дефляции.** Последовательная дефляция постепенно теряет ортогональность компонент при больших `k`. Флаг `reorthogonalize` (по умолчанию `false`) включает Gram–Schmidt новой `t`/`p` против всех ранее найденных. Включать при `k`, близком к рангу, или при требовании строгой ортогональности.
- **Вырождение `tᵀt ≈ 0`.** Если на старте/в ходе итераций `tᵀt` ниже порога `eps` — компонента вырождена (данные исчерпаны по рангу): прекращаем и возвращаем уже найденные компоненты, число которых может быть `< k`.
- **Несходимость.** Если `max_iter` достигнут без выполнения критерия `tol` — это не ошибка по умолчанию: возвращаем последнюю `t/p` и помечаем в `iterations`/`converged` диагностике. Опция `strict` может превратить это в `Err`.
- **NaN/Inf на входе** → `Err(NonFiniteValue)` (как в linreg).

### 3.4. Соглашение о знаке

Собственные векторы определены с точностью до знака; без фиксации тесты на воспроизводимость нестабильны. Правило: для каждой `pₐ` находим элемент с максимальным модулем; если он отрицателен — домножаем `pₐ` и `tₐ` на `−1`. Это делает знак детерминированным и не меняет разложение (`tₐpₐᵀ` инвариантно).

## 4. Архитектура Rust-ядра

Принцип общий с linreg: ядро не знает про WASM/JS; чистый код на `Vec<f64>`; `std` ради `Vec` и `f64::sqrt`.

### 4.1. Структура крейта

```
crates/
  pca-core/             # чистое ядро, без wasm
    src/
      lib.rs
      matrix.rs         # операции над списком колонок (column-major), dot/axpy/norm
      stats.rs          # FeatureStats: хранение μ/σ и раскрутка нагрузок (общий стиль с linreg)
      nipals.rs         # извлечение компонент, дефляция, реортогонализация
      model.rs          # Pca: fit / transform / inverse_transform
      metrics.rs        # explained_variance, ratio, reconstruction_error
      error.rs          # типы ошибок (те же, что в linreg)
    tests/
      integration.rs    # см. раздел 7
  pca-wasm/             # обёртка wasm-bindgen, зависит от pca-core
    src/lib.rs
```

### 4.2. Представление данных — список колонок

Идентично linreg-контракту. Вход — `columns: &[&[f64]]`, `columns[j]` — признак `j` длиной `n_rows`; `m = columns.len()`. Инварианты входа: непустой список, равные длины колонок, отсутствие NaN/Inf, `n_rows ≥ 2` (нужна дисперсия). Нарушение → `EmptyDataset` / `DimensionMismatch` / `NonFiniteValue`.

Колоночное хранение естественно для NIPALS: `p = Xᵀt` — это `m` независимых dot-product'ов «остатки · колонка» (последовательный проход по непрерывной памяти), а `t = Xp` и дефляция `X − t·pᵀ` — аккумуляторные проходы по колонкам (`X[:,j] -= t · p[j]`).

### 4.3. Ключевые типы (эскиз API)

```rust
// error.rs — те же варианты, что в linreg
#[derive(Debug, Clone, PartialEq)]
pub enum PcaError {
    DimensionMismatch { expected: usize, got: usize },
    EmptyDataset,
    NotFitted,
    InvalidHyperparameter(String),
    NonFiniteValue,
    NotConverged { component: usize, iterations: usize }, // только при strict=true
}

pub struct Config {
    pub n_components: usize,    // k; 0 == min(n_rows-1, m) (default 0 → авто)
    pub tol: f64,              // сходимость score-вектора, default 1e-9
    pub max_iter: usize,       // на компоненту, default 500
    pub reorthogonalize: bool, // Gram–Schmidt против найденных, default false
    pub strict: bool,         // несходимость → Err, default false
}

impl Default for Config { /* значения выше */ }

// stats.rs — общий стиль с linreg (вариант B): μ/σ только для раскрутки.
pub struct FeatureStats { means: Vec<f64>, stds: Vec<f64> }
impl FeatureStats {
    pub fn new(means: Vec<f64>, stds: Vec<f64>) -> Result<Self, PcaError>;
}

pub struct Pca {
    loadings: Vec<f64>,        // P, row-major: k строк по m (компонента за компонентой)
    scores: Vec<f64>,          // T обучающих данных, k строк по n (кэш из fit)
    explained_variance: Vec<f64>,        // λₐ, длина k
    explained_variance_ratio: Vec<f64>,  // λₐ / total_var
    singular_values: Vec<f64>,           // ‖tₐ‖
    n_components: usize,       // фактически извлечено (может быть < запрошенного k)
    n_features: usize,
    iterations: Vec<usize>,    // число итераций на компоненту (диагностика)
    stats: Option<FeatureStats>,
    config: Config,
}

impl Pca {
    pub fn new(config: Config) -> Result<Self, PcaError>; // валидирует гиперпараметры

    // μ/σ — опционально, ТОЛЬКО для раскрутки нагрузок в исходное пространство.
    pub fn set_feature_stats(&mut self, means: Vec<f64>, stds: Vec<f64>)
        -> Result<(), PcaError>;

    // КОНТРАКТ: columns — УЖЕ ЦЕНТРИРОВАННЫЕ (вариант B) колонки.
    // Вычисляет P, объяснённую дисперсию и кэширует scores T обучающих данных.
    pub fn fit(&mut self, columns: &[&[f64]]) -> Result<(), PcaError>;

    // Проекция данных в пространство компонент: X_centered · Pᵀ → scores (n × k).
    // columns должны быть в ТОМ ЖЕ масштабе/центровке, что и на fit.
    pub fn transform(&self, columns: &[&[f64]]) -> Result<Vec<Vec<f64>>, PcaError>;

    // Обратная реконструкция: scores · P → приближение центрированных колонок.
    pub fn inverse_transform(&self, scores: &[&[f64]]) -> Result<Vec<Vec<f64>>, PcaError>;

    // Геттеры (NotFitted до fit):
    pub fn loadings(&self) -> Result<&[f64], PcaError>;        // P, k×m
    pub fn scores(&self) -> Result<&[f64], PcaError>;          // T обуч. данных, k×n
    pub fn explained_variance(&self) -> &[f64];
    pub fn explained_variance_ratio(&self) -> &[f64];
    pub fn singular_values(&self) -> &[f64];
    pub fn n_components(&self) -> usize;

    // Нагрузки, раскрученные в исходное (нестандартизованное) пространство, если задан σ.
    // Без μ/σ возвращает loadings как есть (в стандартизованном пространстве).
    pub fn loadings_original_space(&self) -> Vec<f64>;
}
```

> Как и в linreg (вариант B): `fit`/`transform` работают над поданными (центрированными) колонками и μ/σ не применяют. `set_feature_stats` влияет только на `loadings_original_space()`. Ответственность за единый масштаб/центровку между `fit` и `transform` — на вызывающей стороне; ядро это не проверяет.

### 4.4. Раскрутка нагрузок в исходное пространство

Если перед центрированием данные ещё и стандартизовались (делёж на `σⱼ` на стороне JS), то нагрузки в исходных единицах признаков:

```
p_origₐⱼ = pₐⱼ / σⱼ
```

(μ на нагрузки не влияет — центрирование сдвигает начало координат, но не направления.) Если стандартизации не было (`σ` не задан) — нагрузки уже в исходном пространстве.

## 5. Алгоритм обучения (псевдокод)

```
fit(columns):                                  # columns: УЖЕ ЦЕНТРИРОВАННЫЕ колонки
    m = columns.len();  n = columns[0].len()
    валидировать инварианты входа (4.2) → EmptyDataset / DimensionMismatch
    проверить NaN/Inf → NonFiniteValue
    k = config.n_components or min(n-1, m)
    total_var = Σⱼ ( Σᵢ columns[j][i]² ) / (n-1)     # колонки центрированы ⇒ это дисперсии
    X = копия columns в рабочий буфер (column-major)
    P = []; T = []; ev = []; sv = []; iters = []
    для a в 1..=k:
        rem = Σⱼ Var(X[:,j])
        если rem < eps: break                         # ранг исчерпан
        # --- NIPALS на текущей X ---
        t = X[:, argmax_j Var(X[:,j])]                # старт
        для it в 1..=max_iter:
            # p = Xᵀ t / (tᵀ t): m независимых dot-product'ов по колонкам
            tt = dot(t,t)
            для j: p[j] = dot(X[:,j], t) / tt
            p = p / ‖p‖
            # t_new = X p: аккумулятор по колонкам
            t_new = zeros(n)
            для j: axpy(t_new, p[j], X[:,j])          # t_new += p[j]·X[:,j]
            если ‖t_new − t‖/‖t_new‖ < tol: t = t_new; converged=true; break
            t = t_new
        если !converged и strict: return Err(NotConverged{a, max_iter})
        если reorthogonalize:
            gram_schmidt(t, против всех T);  gram_schmidt(p, против всех P)
        fix_sign(p, t)                                # п. 3.4
        ev.push(dot(t,t)/(n-1)); sv.push(sqrt(dot(t,t)))
        P.push(p); T.push(t); iters.push(it)
        # --- дефляция: X[:,j] -= t · p[j] ---
        для j: axpy(X[:,j], −p[j], t)
    сохранить P,T,ev,sv,iters; ratio = ev / total_var; n_components = len(P)
```

> Все три горячие операции (`Xᵀt`, `Xp`, дефляция) организованы как внешний цикл по колонкам и внутренний по строкам — последовательный доступ к памяти при колоночном хранении. Ковариационная матрица `m×m` не формируется никогда: выигрыш при `m`, сопоставимом или большем `n`.

## 6. WASM-бэкенд

Тонкая обёртка; ядро остаётся чистым. Граница идентична linreg: список колонок передаётся как один плоский `Float64Array` (конкатенация колонок) + `n_rows`, на стороне Rust `chunks_exact(n_rows)` восстанавливает срезы колонок без копий.

```rust
// pca-wasm/src/lib.rs
use wasm_bindgen::prelude::*;
use pca_core::{Pca, Config};

#[wasm_bindgen]
pub struct WasmPca { inner: Pca }

#[wasm_bindgen]
impl WasmPca {
    #[wasm_bindgen(constructor)]
    pub fn new(n_components: usize, tol: f64, max_iter: usize) -> Result<WasmPca, JsError> {
        let cfg = Config { n_components, tol, max_iter, ..Default::default() };
        Ok(Self { inner: Pca::new(cfg).map_err(to_js)? })
    }

    // μ/σ — опционально, только для раскрутки нагрузок (вариант B).
    pub fn set_feature_stats(&mut self, means: &[f64], stds: &[f64]) -> Result<(), JsError> {
        self.inner.set_feature_stats(means.to_vec(), stds.to_vec()).map_err(to_js)
    }

    // flat_columns: конкатенация центрированных колонок; n_rows для реконструкции срезов.
    pub fn fit(&mut self, flat_columns: &[f64], n_rows: usize) -> Result<(), JsError> {
        let cols = to_columns(flat_columns, n_rows).map_err(to_js)?;
        self.inner.fit(&cols).map_err(to_js)
    }

    // Возврат плоского row-major (k × n_rows) scores; форму k/n JS знает из вызова.
    pub fn transform(&self, flat_columns: &[f64], n_rows: usize) -> Result<Vec<f64>, JsError> {
        let cols = to_columns(flat_columns, n_rows).map_err(to_js)?;
        Ok(flatten(self.inner.transform(&cols).map_err(to_js)?))
    }

    pub fn loadings(&self) -> Result<Vec<f64>, JsError> {            // k×m плоско
        self.inner.loadings().map(|s| s.to_vec()).map_err(to_js)
    }
    pub fn explained_variance_ratio(&self) -> Vec<f64> {
        self.inner.explained_variance_ratio().to_vec()
    }
    pub fn n_components(&self) -> usize { self.inner.n_components() }
}

fn to_columns(flat: &[f64], n_rows: usize) -> Result<Vec<&[f64]>, PcaError> {
    if n_rows == 0 || flat.len() % n_rows != 0 { /* -> DimensionMismatch */ }
    Ok(flat.chunks_exact(n_rows).collect())
}
```

Сборка: `wasm-pack build crates/pca-wasm --target web`. Замечания по копиям/материализации `Vec<f64>` → `Float64Array` те же, что в linreg.

## 7. Валидационные тесты

Структура и стиль как в linreg: (а) unit-тесты, (б) числовые тесты корректности, (в) property-тесты, (г) тесты WASM-границы. Эталоны — аналитические или через известные свойства PCA; без внешних ML-зависимостей.

### 7.1. Unit-тесты компонентов

| # | Что проверяем | Ожидание |
|---|---|---|
| U1 | `matrix`: dot/axpy/norm на известных векторах | совпадают с ручным расчётом |
| U2 | `FeatureStats::new` с валидными μ/σ | конструируется; σ≈0 → 1.0 |
| U3 | `FeatureStats::new` с σ<0 или NaN/Inf | `Err(InvalidHyperparameter)` / `Err(NonFiniteValue)` |
| U4 | `fix_sign`: ведущий по модулю элемент `p` сделан положительным | знак детерминирован, `t·pᵀ` неизменно |
| U5 | Валидация: `tol≤0`, `max_iter=0`, `n_components>m` | `Err(InvalidHyperparameter)` |
| U6 | NaN/Inf во входных колонках | `Err(NonFiniteValue)` |
| U7 | Колонки разной длины | `Err(DimensionMismatch)` |
| U8 | Пустой список / `n_rows<2` | `Err(EmptyDataset)` |
| U9 | Любой геттер (`loadings`,`scores`) до `fit` | `Err(NotFitted)` |
| U10 | Контракт колонок: `columns[j]` = признак `j` | нагрузки соответствуют поданным колонкам |

### 7.2. Корректность PCA (числовые тесты)

| # | Сценарий | Ожидание |
|---|---|---|
| C1 | **Известные собственные значения 2×2**: диагональная ковариация (центрированные данные с дисперсиями 4 и 1) | `explained_variance ≈ [4, 1]` (tol 1e-6); `P ≈ единичные оси` |
| C2 | **Ортонормированность нагрузок**: `P·Pᵀ ≈ I` | внедиагональные < 1e-8, диагональ ≈ 1 |
| C3 | **Ортогональность scores**: `tₐᵀ·t_b ≈ 0` при `a≠b` | < 1e-6 относительно `‖tₐ‖‖t_b‖` |
| C4 | **Доли дисперсии суммируются**: Σ ratio при `k=rank` | ≈ 1.0 (tol 1e-9) |
| C5 | **Реконструкция при полном `k`**: `T·P` восстанавливает центрированный `X` | макс. поэлементная ошибка < 1e-8 |
| C6 | **Усечённая реконструкция**: `k < rank`, ошибка реконструкции | равна сумме отброшенных собственных значений (tol 1e-6) |
| C7 | **Сонаправленность первой компоненты**: данные вдоль прямой `y=2x` (+малый шум поперёк) | `p₁` коллинеарна `(1,2)/‖·‖` (tol 1e-3) |
| C8 | **Сверка с эталонным разложением** известной 3×3 ковариации | `explained_variance` и `P` совпадают с заранее посчитанными собственными значениями/векторами (tol 1e-6, с учётом знака по 3.4) |
| C9 | **Раскрутка нагрузок**: стандартизованный вход + σ | `loadings_original_space()` = `pₐⱼ/σⱼ` (tol 1e-9) |
| C10 | **transform на обучающих данных** = кэшированные `scores()` | совпадение (tol 1e-9) |
| C11 | **Ранговая недостаточность**: `m` колонок, но ранг `r<m` (дублированные признаки) | извлекается ровно `r` компонент, остаток отсечён по `eps` |
| C12 | **Реортогонализация улучшает ортогональность**: большой `k`, `reorthogonalize=false` vs `true` | при `true` отклонение `P·Pᵀ` от `I` строго меньше |
| C13 | **Детерминизм**: два `fit` на одних данных | побайтово идентичные `P`, `T`, дисперсии |
| C14 | **Несходимость**: жёсткий `tol`, малый `max_iter`, `strict=true` | `Err(NotConverged{..})`; при `strict=false` — результат + заполненный `iterations` |

### 7.3. Property-based тесты (`proptest`)

| # | Свойство |
|---|---|
| P1 | Для случайных центрированных данных нагрузки ортонормированы (`P·Pᵀ≈I`) при любом `k≤rank` |
| P2 | Объяснённая дисперсия невозрастающая по компонентам: `ev[a] ≥ ev[a+1]` |
| P3 | `0 ≤ ratio[a] ≤ 1` и `Σ ratio ≤ 1 + 1e-9` |
| P4 | При полном `k` ошибка реконструкции `‖X − T·Pᵀ‖` < ε |
| P5 | `transform` не возвращает NaN/Inf на любых конечных входах после успешного `fit` |

### 7.4. Тесты WASM-границы

Запуск через `wasm-bindgen-test` (headless-браузер/Node).

| # | Что проверяем |
|---|---|
| W1 | Плоская конкатенация колонок + `n_rows` корректно читается; `fit` отрабатывает |
| W2 | `transform`/`loadings` возвращают `Float64Array` правильной длины (`k·n`, `k·m`) |
| W3 | Ошибки ядра прокидываются как `JsError`/исключение JS |
| W4 | Числовой паритет: один датасет через нативный `cargo test` и через WASM — совпадающие `P`/дисперсии (tol 1e-9) |
| W5 | `set_feature_stats` влияет только на раскрутку нагрузок, не на `transform` |

### 7.5. Эталонная сверка (опционально, отдельной задачей CI)

Сгенерировать небольшой датасет, прогнать офлайн `numpy.linalg.eigh`/`sklearn.decomposition.PCA`, сохранить ожидаемые собственные значения и векторы (с фиксацией знака по 3.4) в JSON-фикстуру и сверять с допуском. Это артефакт теста, а не зависимость кода.

## 8. Гиперпараметры по умолчанию (сводка)

| Параметр | Default | Обоснование |
|---|---|---|
| `n_components` | `0` → `min(n−1, m)` | по умолчанию полный набор компонент |
| `tol` | `1e-9` | сходимость score-вектора |
| `max_iter` | `500` | потолок итераций на компоненту |
| `reorthogonalize` | `false` | скорость; включать при большом `k` или требовании строгой ортогональности |
| `strict` | `false` | несходимость не считается ошибкой по умолчанию |

Центрирование/масштабирование выполняется вне ядра (вариант B). μ/σ задаются опционально через `set_feature_stats` и используются только для `loadings_original_space()`.

## 9. Этапы разработки

1. **Каркас** — два крейта, `Config::default`, типы ошибок, заглушки API. Тесты U5–U9.
2. **Матрица + FeatureStats** — dot/axpy/norm над колонками; `FeatureStats::new` с валидацией. Тесты U1–U3, U10.
3. **NIPALS на одну компоненту** — итерация `p`/`t`, критерий `tol`, `fix_sign`. Тесты U4, C7.
4. **Цикл с дефляцией** — извлечение `k` компонент, объяснённая дисперсия, сингулярные значения. Тесты C1–C4, C8.
5. **Реконструкция** — `transform`, `inverse_transform`, кэш scores. Тесты C5–C6, C10.
6. **Раскрутка + раноговые случаи** — `loadings_original_space`, отсечение по `eps`. Тесты C9, C11.
7. **Устойчивость** — реортогонализация, несходимость/`strict`, детерминизм. Тесты C12–C14.
8. **Property-тесты.** P1–P5.
9. **WASM-обёртка** + `wasm-pack`: плоская конкатенация, реконструкция `chunks_exact`. Тесты W1–W5.
10. **CI** — `cargo test`, `wasm-pack test`, опциональная эталонная фикстура (7.5).
