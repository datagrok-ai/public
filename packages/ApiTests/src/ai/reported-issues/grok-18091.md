# GROK-18091: PC plot transformation persistence across filter panel toggles

Regression coverage for a PC plot bug fixed in 1.26.0/1.26.3: setting a
`transformation` while the filter panel was open caused the PC plot to go
empty after the filter panel was closed. The fix preserves the transformed
dataset across filter-panel toggles so the viewer remains functional.

## Triage scope

`IPcPlotSettings.transformation` is a JSON-serialized string (see
`public/js-api/src/interfaces/d4.ts:2748`). The viewer treats it as an opaque
payload, so these tests round-trip the **exact** string rather than parsing
it. The original symptom requires the filter panel to be visible during the
mutation, so case 2 attaches a `TableView`, opens its filter group via
`tv.getFiltersGroup({createDefaultFilters: false})`, applies the
transformation, then closes the group via `fg.setActive(false)` — the API
present on `FilterGroup` per `public/js-api/src/viewer.ts:546`.

## Cases

| # | Case                                                                                   | What it pins                                                                                |
|---|----------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------|
| 1 | Detached `DG.Viewer.pcPlot(df, {columnNames})`, `setOptions({transformation})` round-trip | `look.transformation` round-trips the exact JSON string (no parse, no normalisation).       |
| 2 | Attached PC plot on a `TableView`, open filter group, set transformation, close filter group | `look.columnNames` and `look.transformation` survive the open/close toggle; viewer remains. |
| 3 | `setOptions({transformation: ''})` clears the field                                    | `look.transformation === ''` after clearing.                                                 |

## API surface exercised

- `DG.Viewer.pcPlot(df, {columnNames})` (`viewer.ts:275`)
- `IPcPlotSettings.transformation: string` (`interfaces/d4.ts:2748`)
- `viewer.setOptions(...)` and `viewer.getOptions(true).look`
- `TableView.getFiltersGroup({createDefaultFilters: false})` (`views/view.ts:405`)
- `FilterGroup.setActive(active)` (`viewer.ts:546`)

## Out of scope

- The exact JSON shape the transformation pipeline accepts at runtime. We
  pin a small JSON-string payload via `setOptions` and read it back via
  `getOptions(true).look` — the round-trip is the regression invariant the
  fix scopes to.
- UI filter panel docking / sidebar layout. The bug surface we exercise is
  the JS-reachable open/close via `getFiltersGroup` + `setActive(false)`.
- `viewer.dataFrame === df` identity (round-5: unreliable).

## Risks / unverifiable

- `transformation` is a JSON-serialized opaque string. The test stores a
  small JSON object literal as a string (e.g. `'{}'` and a non-trivial
  object) and asserts string equality on the round-trip. Real
  transformations may serialize to canonically different strings if the
  viewer rewrites them; if a future build normalises whitespace or key
  order this test will surface that.
- The detached `DG.Viewer.pcPlot(df, ...)` is never attached to a view in
  case 1, so `viewer.close()` is skipped there per the round-5 finding
  that closing a never-attached viewer can throw. Case 2 attaches via
  `tv.addViewer(DG.VIEWER.PC_PLOT, ...)` and cleans up via `tv.close()` in
  a `try/finally`.
- `FilterGroup.setActive(false)` is the JS-reachable equivalent of
  closing the filter panel; if the platform diverges (e.g. requires a
  separate dispose call) we will need to revisit.
