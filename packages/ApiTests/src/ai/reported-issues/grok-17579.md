# GROK-17579: Bar chart tooltip hide

Regression coverage for the bug fixed in 1.24.0: right-click on a Bar chart →
**Tooltip > Hide** did not suppress the tooltip, while the same menu worked on
every other viewer.

## Triage scope

The user-visible symptom is canvas-dependent (mouse-over a bar; tooltip should
not appear). The JS API has no hook for "is the tooltip currently rendered",
so a pixel-accurate repro belongs in the manual / image-diff suite. These tests
deliberately limit themselves to the **state machine integrity** that the
fix had to wire up: the bar chart must expose a `showTooltip` property, the
"hide" enum value must persist through the property bag, and every advertised
choice must survive a `setOptions` / `getOptions` round trip. This is a
**fix-stuck guard** — if the property gets unregistered or the enum is renamed
again, this category fails.

## Cases

| # | Case                                                       | What it pins                                                                |
|---|------------------------------------------------------------|-----------------------------------------------------------------------------|
| 1 | `showTooltip` appears in `viewer.getProperties()`          | The property is registered on the bar chart's look class with `choices`.    |
| 2 | Set the "hide" choice via `viewer.props['showTooltip']`    | `props` and `getOptions(true).look.showTooltip` both reflect the new value. |
| 3 | Iterate every value from `prop.choices` and round-trip it  | None of the documented choices are silently rejected.                       |

The "hide" choice is discovered at runtime: case 2 looks for the literal
`'do not show'` first (matching the convention used on Scatterplot / Box plot
/ Grid in `*_look.dart`), then falls back to any choice whose lowercased text
mentions `not`, `hide`, `off`, or equals `never`. The test asserts that *some*
hide value exists and round-trips — it does not hard-code the spelling.

## API surface exercised

- `DG.Viewer.barChart(df, settings)`
- `viewer.props['showTooltip']` (read/write through the property bag)
- `viewer.getProperties()` → `Property.name`, `Property.choices`
- `viewer.getOptions(true).look`

## Out of scope

- Whether the tooltip actually disappears on the canvas (visual-only).
- Right-click menu plumbing (`onContextMenu` → "Tooltip > Hide" item) — that is
  Dart-side wiring and is exercised by the manual tutorial under
  `core/client/d4/lib/src/viewers/bar_chart/`.
- The reverse direction (custom row tooltip column-list editing). `rowTooltip`
  is a different property; the original ticket is only about the `Hide` action.
