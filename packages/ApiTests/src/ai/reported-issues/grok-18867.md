# GROK-18867: PC plot colorMin/colorMax persistence

Regression coverage for the PC plot defect where the per-column range
slider min/max could be reset incorrectly when the active columns set
changed. Shipped in 1.26.3. The internal slider state is not part of the
public JS API surface, so we pin the JS-API-visible analog: the
`colorMin` and `colorMax` keys on `IPcPlotSettings`
(`public/js-api/src/interfaces/d4.ts:2716-2718`).

## Triage scope

We pin a *tolerant coherence invariant* on the look bag after a
`columnNames` change: either both color bounds remain numerically
preserved at the values we wrote, or both are cleared together. The bug
was the asymmetric, half-set state (one numeric, the other
null/undefined). Per-column slider internals and visual range pinning are
out of scope.

## Cases

| # | Case                                                                                  | What it pins                                                                                  |
|---|---------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------|
| 1 | `DG.Viewer.pcPlot(df, ...)`; `setOptions({colorColumnName, colorMin: 10, colorMax: 90})` | Both keys are present in `getOptions(true).look` and equal 10/90. Round-trip baseline.        |
| 2 | After case 1, `setOptions({columnNames: ['age','height']})`                            | Look bag: `(colorMin === 10 && colorMax === 90)` OR `(colorMin == null && colorMax == null)`. Half-set is the bug. |
| 3 | Smoke: `setOptions({columnNames: ['height']})` while bounds are set                    | Does not throw; same coherence invariant on `look.colorMin`/`colorMax`.                       |

## API surface exercised

- `DG.Viewer.pcPlot(df, options): DG.PcPlot`
- `IPcPlotSettings.{colorMin, colorMax, colorColumnName, columnNames}`
- `Viewer.setOptions(...)`, `Viewer.getOptions(true).look`

## Out of scope

- Per-column range slider internal state (not on the public interface).
- `viewer.dataFrame === df` identity (unreliable across the interop
  boundary; not asserted here).
- Visual / pixel verification of the color legend or range indicator.

## Risks and unverifiable symbols

- The *exact* post-change semantics — preserve vs. clear — are not part
  of the public contract; we only assert coherence.
- `colorMin` / `colorMax` typing is `number` (non-nullable) on
  `IPcPlotSettings`, but at runtime the look bag may serialize a missing
  bound as `null` or `undefined`. The test uses `== null` to cover both.
