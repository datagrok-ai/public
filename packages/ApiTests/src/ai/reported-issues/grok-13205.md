# GROK-13205: Trellis row source variants

Regression coverage for trellis-plot inner-viewer row-source semantics.

## Triage scope

The original bug: trellis-plot inner viewers (Line / Box / Pie / Bar) duplicated
the parent dataframe's data instead of partitioning it across cells when
`rowSource` was set to one of the per-row variants — `Selected`,
`SelectedOrCurrent`, `FilteredSelected`, `MouseOverGroup`, `CurrentRow`,
`MouseOverRow`. Visual partition correctness is the actual symptom.

These tests deliberately limit themselves to **state machine integrity**: each
RowSet literal can be assigned to `viewer.props.rowSource` without throwing,
the value persists through `getOptions(true).look`, and swapping `viewerType`
between the four inner types under each rowSource does not blow away the prop.

Pixel-level partition correctness — i.e., that the inner viewer for cell `(a,
x)` actually shows only the rows matching that combination — is **not asserted
here** and must be validated in manual / image-diff QA.

## Cases

| # | Case                                                                                              | What it pins                                                  |
|---|---------------------------------------------------------------------------------------------------|---------------------------------------------------------------|
| 1 | For each of the six per-row `RowSet` literals, set the corresponding df state and assign `rowSource` | `props.rowSource` and `getOptions(true).look.rowSource` agree |
| 2 | For each rowSource, swap `viewerType` between Bar / Box / Pie / Line                              | Inner-type swap doesn't throw and doesn't clobber `rowSource` |

The tested literals come from `RowSet` in
`public/js-api/src/interfaces/d4.d.ts`: `Selected`, `SelectedOrCurrent`,
`CurrentRow`, `MouseOverRow`, `FilteredSelected`, `MouseOverGroup`. `All` and
`Filtered` are not in scope of this regression (they were not part of the bug
report).

## API surface exercised

- `DG.Viewer.trellisPlot(df, settings)`
- `ITrellisPlotSettings.{viewerType, xColumnNames, yColumnNames, rowSource}`
- `DG.RowSet` enum literal strings
- `df.selection` (BitSet `setAll` / `set`), `df.filter`, `df.currentRowIdx`,
  `df.mouseOverRowIdx`
- `viewer.props[...]`, `viewer.getOptions(true).look`

## Out of scope / known gaps

- **Visual partition correctness is not asserted.** This is the load-bearing
  caveat — JS API can confirm the property bag is wired correctly but cannot
  see whether the inner viewer actually filtered its rows.
- First-paint geometry is intentionally avoided. Viewers are constructed
  *detached* (no `TableView`) and we never call `close()` on a detached viewer.
- `MouseOverGroup` triggers off platform mouse state that the JS API cannot
  fully simulate; we only set `mouseOverRowIdx` so the prop assignment has
  consistent state to chew on.
