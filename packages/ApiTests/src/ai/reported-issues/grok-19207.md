# GROK-19207: Line chart annotation regions after delete

Regression coverage for a line-chart bug fixed in 1.27.0: drawing an
annotation region, then closing/deleting the chart and performing follow-up
actions (re-opening a new chart on the same `TableView`) threw
"Index out of range".

## Triage scope

The original bug requires UI mouse drawing to reproduce the most direct
"draw region, then close" path. From JS we cannot synthesize that drag
gesture, so these tests pin the **lifecycle invariants** that the fix
guarantees:

1. `enableAnnotationRegionDrawing()` can be invoked on a Line chart that is
   attached to a `TableView`, and closing that viewer (via `viewer.close()`
   on the *attached* viewer — not on a detached `df.plot.line` instance,
   which can throw on close per round-5 finding) leaves the platform in a
   state where a fresh Line chart can be added to the same `tv` without
   throwing.
2. The `viewer.meta.annotationRegions` helper (`viewer.ts:773`,
   `ViewerAnnotationRegionsHelper`) accepts an `AnnotationRegion` via
   `add(...)` and survives the same close/re-open cycle.
3. Two consecutive `df.plot.line(...)` lifecycles on the same dataframe
   complete without console errors, exercising the close path that
   previously corrupted internal indices.

## Cases

| # | Case                                                                | What it pins                                       |
|---|---------------------------------------------------------------------|----------------------------------------------------|
| 1 | Attached Line chart, `enableAnnotationRegionDrawing()`, close, re-add | New `LineChartViewer` is added to the same `tv` without throwing. |
| 2 | Attached Line chart, `meta.annotationRegions.add({...})`, close, re-add | Region is persisted to the look JSON; re-opened chart is a fresh `LineChartViewer`. |
| 3 | Two `df.plot.line` factories created and closed via the attached path in sequence | Both viewers are `LineChartViewer`; second one survives. |

## API surface exercised

- `LineChartViewer.enableAnnotationRegionDrawing` (`viewer.ts:602`)
- `LineChartViewer.disableAnnotationRegionDrawing` (`viewer.ts:607`)
- `Viewer.meta.annotationRegions` → `ViewerAnnotationRegionsHelper`
  (`viewer.ts:773`, base `AnnotationRegionsHelper` in `helpers.ts:143`)
- `tv.addViewer(DG.VIEWER.LINE_CHART, ...)`
- `viewer.close()` on the attached viewer
- `tv.viewers` enumeration

## Out of scope

- UI mouse drawing of the region (cannot be synthesized from JS).
- Pixel rendering, region geometry, region drawing animation.
- `viewer.close()` on a detached `df.plot.line` instance — round-5 found
  that this path can throw independently of the GROK-19207 fix, so
  case 3 routes through `tv` to keep the assertion focused on the
  lifecycle bug at hand.

## Risks / unverifiable

- Region creation from JS uses the helper `add(...)` API rather than the
  UI drag, so the exact internal code path that produced "Index out of
  range" before 1.27.0 may differ subtly. Cases 1-3 still exercise the
  close-after-annotation-state lifecycle that the fix is scoped to.
- `tv.removeViewer` is not present on the JS `TableView`, so removal
  goes through `viewer.close()` on the attached instance.
