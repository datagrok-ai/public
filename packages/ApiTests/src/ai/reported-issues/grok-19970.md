# GROK-19970: Bar/Pie chart filter resets on close

Regression coverage for a bar-chart and pie-chart bug fixed in 1.28.0.

## Triage scope

When a Bar chart or Pie chart was configured with `onClick='Filter'`, clicking
a bar/slice would apply a row filter on the underlying `DataFrame`. Closing
the viewer used to leave that filter in place, "stranding" a filter the user
could no longer reach via UI. The fix triggers a filter reset on viewer
disposal ŌĆö but only when `onClick === 'Filter'`. With `onClick='Select'` the
viewer interacts with `df.selection` instead, and `close()` must not touch
`df.filter`.

## Cases (per category)

The file contains **two** `category(...)` blocks ŌĆö one for Bar chart, one for
Pie chart ŌĆö with the same three cases each:

| # | Case | What it pins |
|---|------|--------------|
| 1 | `onClick='Filter'`, simulate post-click filter via `df.filter.set(0, false)`, then `viewer.close()` | After close, `df.filter.trueCount === df.rowCount` (filter reset). |
| 2 | `onClick='Select'`, simulate `df.filter.set(0, false)`, then `viewer.close()` | `df.filter.trueCount` keeps the value we set (`rowCount - 1`); the close path doesn't touch `df.filter`. |
| 3 | All cases run inside `try/finally`; `tv.close()` always runs at the end. | No `tv` leaks even if assertions fail. |

## Risks / caveats

- `viewer.close()` on a never-attached viewer **throws**. The tests only
  call `close()` on viewers attached via `tv.addViewer(...)`.
- We can't synthesize a click on a bar/slice from the JS API, so we
  substitute the post-click state by calling `df.filter.set(0, false)`
  directly. The fix being tested is the **reset-on-close**, which is
  independent of how the filter got set.
- `viewer.dataFrame === df` is unreliable across the DartŌåÆJS wrapper
  boundary, so we read `df.filter` / `df.selection` directly via `df`.
- `PieChart` has no `resetView()` method ŌĆö we don't rely on one.

## Out of scope

- Pixel-level rendering, actual click-driven filter application ŌĆö the
  click pipeline is UI-only.
- We do not pin the platform's internal filter-reset mechanism (event,
  direct mutation, etc.) ŌĆö only the observable post-close state.
