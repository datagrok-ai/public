# GROK-19480: Bar chart stacking + orientation

Regression coverage for a cluster of bar-chart bugs fixed in 1.27.0.

## Triage scope

The original ticket bundles four sub-issues:

1. Stacking failed to activate when the `value` column was switched to certain
   numeric columns.
2. Negative aggregated values were rendered incorrectly when stacking was active.
3. Legend selection order changed the displayed bars.
4. Vertical orientation showed wrong values and lost the legend.

Several of those manifest as visual rendering glitches that are hard to assert
via the JS API. These tests deliberately limit themselves to **state machine
integrity** — the property bag accepts the documented inputs, persists them
through `getOptions(true).look`, and the viewer instance survives without
throwing. Pixel-level checks belong in the manual / image diff suite.

## Cases

| # | Case                                                          | What it pins                                              |
|---|---------------------------------------------------------------|-----------------------------------------------------------|
| 1 | Build a bar chart with negatives + split + stack via `df.plot.bar` | `stackColumnName` is set; instance is a `BarChartViewer`. |
| 2 | Round-trip `valueColumnName` through two numeric columns      | `props` and `look` both reflect the most recent value.    |
| 3 | Set `orientation: 'vertical'` via `setOptions`                | `look.orientation` preserved.                             |
| 4 | `onResetView` fires when `resetView()` is called              | Race against a 1.5s timeout; signal must arrive.          |

## API surface exercised

- `df.plot.bar({value, split, stack})`
- `DG.Viewer.barChart(df, settings)`
- `IBarChartSettings.{valueColumnName, valueAggrType, stack, stackColumnName, orientation}`
- `BarChartViewer.onResetView`, `BarChartViewer.resetView()`
- `Viewer.getOptions(true).look`

## Out of scope

- Pixel rendering, legend layout, axis labels, geometry. First-paint geometry is
  intentionally avoided.
- We do not call `close()` on detached viewers (only on `TableView` instances we
  attached the viewer to).
