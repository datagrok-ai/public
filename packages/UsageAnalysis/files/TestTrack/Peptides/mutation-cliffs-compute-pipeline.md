# Peptides — Mutation cliffs and cluster statistics compute pipelines

SAR launch triggers two background compute pipelines that aren't directly exposed as a button click: (1) mutation-cliff detection, which compares sequence pairs across Web Workers and feeds the Mutation Cliffs cell glyphs, the Sequence Mutation Cliffs viewer, and the Export Mutation Cliffs table; and (2) per-cluster statistics, which (when sequence-space + MCL clustering is enabled) feeds the Logo Summary Table and the Distribution panel's per-cluster summary. Since neither pipeline has a direct UI trigger, this scenario verifies them indirectly through their rendered outputs — confirming the compute actually ran and produced non-trivial, correctly-shaped data, not just that nothing crashed.

## Setup

1. Start from a clean Datagrok shell — no Peptides analysis or `PeptidesModel` on any open DataFrame, no pre-existing `PeptidesView` TableView. Close any pre-existing peptide-related TableViews from prior test runs and confirm `grok.shell.tableViews` is clear of any view whose active DataFrame carries the `peptidesModel` singleton in `DataFrame.temp` (per the atlas description of `peptides.model` at `src/model.ts#L78`: "Singleton stored in `DataFrame.temp[\"peptidesModel\"]`"). The compute-pipeline assertions in both scenarios depend on a fresh `PeptidesModel.init` invoking the mutation-cliffs worker pool from a cold start, with no cached `mutationCliffs` map on the model.
2. Confirm the Peptides package is loaded and its `@init` `initPeptides` function has completed at least once during the session (per atlas sub_feature `peptides.lifecycle.init` at `src/package.ts#L82` — "initializes singletons MonomerWorks (from Bio:getBioLib), TreeHelper, and lazily-loaded monomer-library / sequence-helper components via `PeptideUtils.loadComponents()`"). The MonomerWorks / sequence-helper instances are prerequisites for splitting aligned sequences into per-position columns, which in turn is the prerequisite for the mutation-cliffs pairwise comparison surface; the workers themselves are spawned by `ParallelMutationCliffs` (per atlas `peptides.compute.parallel-mutation-cliffs` at `src/utils/parallel-mutation-cliffs.ts#L10`) on first call to `findMutations` (atlas `peptides.compute.find-mutations` at `src/utils/algorithms.ts#L33`), which runs inside `startAnalysis` → `model.init`.
3. Load the canonical peptide dataset with a Macromolecule column and at least one numerical activity column — the package's `aligned.csv` sample under `public/packages/Peptides/files/aligned.csv` (FASTA-notation `AlignedSequence` column + `IC50` numerical column). Open the file as a TableView via `grok.data.loadTable` or via the `Peptides` app's `Simple demo` button (per `src/package.ts#L117` — `openDemoData('aligned.csv')`); the resulting TableView's name is `PeptidesView`. Confirm the active DataFrame has a Macromolecule column (`bySemType('Macromolecule') !== null`) and at least one numerical column with no missing values — both are validation gates that `peptidesDialog` (`src/package.ts#L131`) checks before opening the SAR dialog. The fixture's row count should be in the small-to-medium range (single-digit to low-hundreds of sequences) so that the pairwise comparison matrix is large enough to exercise the worker chunking but small enough to complete within typical test timeouts.
4. Reduce to a fast working subset before launching SAR: select the first 200 rows of the loaded table, then **Select > Extract Selected Rows** (backed by the `CmdExtractSelectedRows` platform function). A new TableView opens containing only those 200 rows; **all subsequent Setup and Scenario steps operate on this extracted table.** Rationale: the MCL clustering + Logo Summary Table attach + parallel-mutation-cliffs worker pass all scale with row count — on the full 647-row dataset the SAR-attach-to-LST chain is ~137 s (warm), whereas on the 200-row subset it completes in seconds, so the readiness gates and the outer test budget can be cut substantially. The first 200 rows of this SAR series still carry a dense single-mutation cliff population (2242 single-mutation pairs by static analysis of the source `aligned.csv`), so the mutation-cliffs compute pipeline is exercised on real, non-empty cliffs — Scenario 1 step 3 asserts the aggregated cliffs Map is non-empty (the false-green guard). Scope reduction: on the 200-row subset MCL may cluster into a single group; when it does, Scenario 2's multi-cluster row-enumeration (`>= 2` relaxed to `>= 1`) and the cluster-switch "not stale" delta are recorded as a scope reduction (the per-cluster stats + Distribution summary are still asserted on the single cluster) — multi-cluster switching remains covered on the full 647-row dataset.

## Scenarios

### Scenario 1 — SAR launch runs the mutation-cliffs compute pipeline end-to-end; its three consumers (cell glyphs, the dedicated viewer, and Export) all show non-trivial data

The mutation-cliffs pipeline is not exposed via a direct user gesture — it runs as a side effect of the SAR launch, chunking pairwise sequence comparisons across Web Workers and aggregating the qualifying mutation-cliff pairs. This scenario checks it ran correctly by inspecting its three downstream consumers: (a) the Mutation Cliffs cell glyphs on the Sequence Variability Map render non-empty circles, (b) the dedicated Sequence Mutation Cliffs viewer's line chart shows non-trivial data for a chosen position, and (c) the Export Mutation Cliffs table contains at least one row.

1. With the FASTA peptide dataset loaded as `PeptidesView` (per Setup step 3), invoke `Bio | Analyze | SAR...` from the top menu (atlas sub_feature `peptides.workflow.sar-dialog`, registered at `src/package.ts#L131` as `peptidesDialog` → `Bio Peptides`). Confirm the `analyzePeptidesUI` config dialog opens (per atlas sub_feature `peptides.workflow.analyze-ui` at `src/widgets/peptides.ts#L26`).
2. In the SAR config dialog, ensure the Sequence Variability Map (MonomerPosition) viewer toggle is ON (drives `showMonomerPosition: true` → the MonomerPosition viewer is added via `model.ts#L1234`'s `addMonomerPosition()`, which is the host viewer for the Mutation-Cliffs cell glyphs). Default scaling (`-lg`) is fine. Click OK; allow `startAnalysis` (per atlas sub_feature `peptides.workflow.start-analysis` at `src/widgets/peptides.ts#L248`) to complete. Watch for any console errors or unhandled-promise rejections during the launch — the worker spawn + worker-message round-trip is the critical surface this scenario asserts on.
3. Confirm the mutation-cliffs compute pipeline ran end-to-end by inspecting the model's cached state. Read `PeptidesModel.getInstance(table)` and confirm: (a) `model.mutationCliffs` is non-null and has at least one entry (the outer Map keyed by monomer, inner Map keyed by position, inner-inner array of (index1, index2) pairs per the atlas description of `peptides.compute.find-mutations`'s output); (b) `model.monomerPositionStats` (computed via `calculateCliffsStatistics` per atlas) is non-null with non-empty entries for at least one (monomer, position) combination — each entry contains `meanDifference`, `pValue`, `count`, `ratio` per atlas description of `peptides.compute.get-summary-stats` at `src/utils/algorithms.ts#L179`. The presence of these cached structures confirms that (i) `extractColInfo` packed the position columns and the activity column into the worker-friendly raw form (per atlas `peptides.util.extract-col-info`), (ii) `findMutations` chunked the upper-triangular comparison matrix and dispatched chunks to the worker pool, (iii) each worker (`peptides.compute.mutation-cliffs-worker`) applied the activity-delta + max-mutation-count thresholds and returned qualifying (position, index1, index2) triples, (iv) `ParallelMutationCliffs` aggregated the worker outputs into the model's `mutationCliffs` map, and (v) `calculateCliffsStatistics` derived the per-(monomer, position) summary stats from the aggregated raw cliff map.
4. Switch the MonomerPosition viewer (Sequence Variability Map) to Mutation Cliffs mode via the SAR viewer's context menu — right-click the viewer, expand the `Mode` submenu, and click `Mutation Cliffs` (atlas `peptides.viewers.sar-base` at `src/viewers/sar-viewer.ts#L104`: "Implements context menu including Export actions"; the mode switch is part of the dual-mode behavior documented at `peptides.viewers.monomer-position` — "Two modes: Invariant Map (cells colored by aggregated stats) and Mutation Cliffs (circle size = count, color = mean diff)"). After mode switch, the cell renderer is the Mutation-Cliffs glyph (atlas `peptides.rendering.mutation-cliffs-cell` at `src/utils/cell-renderer.ts#L52`). Visually confirm that ≥ 1 cell in the heatmap renders a non-empty colored circle (count > 0). The cell renderer reads from `model.mutationCliffsMaskInfo` — the per-cell mask projected from the nested `mutationCliffs` map by `mutationCliffsToMaskInfo` (atlas `peptides.util.mutation-cliffs-to-mask-info` at `src/utils/misc.ts#L443`). The presence of any non-empty circle is the UI-side assertion that `mutationCliffsToMaskInfo` ran correctly post-compute. (If ALL cells render empty after compute completed and `model.mutationCliffs` is non-empty, the assertion fails — this isolates a regression in the projection step from a regression in the compute step.)
5. Add the dedicated `Sequence Mutation Cliffs` viewer to the analysis view via `Add viewer | Sequence Mutation Cliffs` (atlas `peptides.viewers.mutation-cliffs` at `src/package.ts#L205`, atlas interaction "Add viewer | Sequence Mutation Cliffs"). Confirm via `PeptidesModel.getInstance(table).findViewer(VIEWER_TYPE.MUTATION_CLIFFS)` that the viewer is non-null. The viewer is a `DG.JsViewer` (`MutationCliffsViewer`) whose root contains a line chart of mutation cliffs at a selected position; per atlas description "Splits by series column; syncs selection with the main DataFrame". Pick a position with non-empty cliff data (per Step 4, any column whose header WebLogo had at least one populated Mutation-Cliffs cell qualifies — click the column header to set the viewer's selected position) and confirm the line chart contains non-trivial trace data (≥ 1 series with ≥ 1 plotted point). This is the second downstream-consumer assertion that the compute pipeline's `mutationCliffs` map is correctly consumed by `MutationCliffsViewer`.
6. Trigger the Export → Export Mutation Cliffs action via the SAR viewer's right-click context menu (atlas `peptides.viewers.sar-base.export-mutation-cliffs` at `src/viewers/sar-viewer.ts#L560`, atlas interaction "right-click SAR viewer → Export → Export Mutation Cliffs"). The column-picker dialog opens; accept default columns and click OK. A new TableView opens containing one row per unique mutation-cliff pair (per atlas description: "Seq 1, Seq 2, Mutation (semtype `MacromoleculeDifference`, format `seq1#seq2`), Seq 1/Seq 2 activity, Delta, plus paired columns for each chosen extra column"). Confirm the exported TableView's DataFrame has ≥ 1 row. This is the third downstream-consumer assertion that the compute pipeline's output survived end-to-end: from worker → `ParallelMutationCliffs` aggregation → `model.mutationCliffs` → `calculateCliffsStatistics` (whose summary stats are not directly exported but whose presence is the precondition for the export-button's enablement state) → `SARViewer.exportMutationCliffs` traversal of the cliff map → emitted TableView row construction.
7. Confirm no null-receiver, worker-message-deserialization, or worker-spawn-failure console errors fired during the SAR launch (steps 2-3) or during the mode switch + viewer add + export (steps 4-6). The pipeline's surface is the worker boundary (postMessage / onmessage round-trip); regressions here surface as either missing data (the cached `mutationCliffs` map is empty when it should not be) or as console errors. The combined assertion: (a) the cached compute outputs are non-empty per Step 3, AND (b) the three downstream consumers (cell renderer, viewer, exporter) all show non-trivial data per Steps 4-6, AND (c) no console errors fired throughout.

Expected (assertion summary):
- After `startAnalysis` completes, `PeptidesModel.getInstance(table).mutationCliffs` is non-null and contains ≥ 1 entry, and `model.monomerPositionStats` (derived via `calculateCliffsStatistics` + `getSummaryStats`) is non-empty.
- Mode switch to Mutation Cliffs renders ≥ 1 non-empty circle glyph in the MonomerPosition heatmap, confirming `mutationCliffsToMaskInfo` projected the nested cliff map into the per-cell mask consumed by the cell renderer.
- The dedicated `Sequence Mutation Cliffs` viewer (after Add viewer | Sequence Mutation Cliffs + position selection) renders ≥ 1 series of ≥ 1 plotted point, confirming `MutationCliffsViewer` consumed the same `mutationCliffs` map.
- The Export → Export Mutation Cliffs TableView contains ≥ 1 row per unique cliff pair, confirming the cliff map survived round-trip into the exported TableView.
- No null-receiver, worker-deserialization, or worker-spawn-failure console errors fire throughout the SAR launch + mode switch + viewer add + export chain.

### Scenario 2 — Sequence-space + MCL clustering feeds per-cluster statistics into the Logo Summary Table and the Distribution panel

Complements Scenario 1 by exercising the second background compute path: when sequence-space dim-reduction and MCL clustering are enabled, the resulting clusters column feeds a per-cluster statistics pass that populates the Logo Summary Table's summary grid (members, mean difference, p-value) and the Distribution panel's per-cluster summary record. This scenario checks both consumers show non-trivial, cluster-specific values, and that selecting a different cluster updates the Distribution summary rather than leaving it stale.

1. With the SAR layout from Scenario 1's Setup attached (or a fresh launch), invoke `Bio | Analyze | SAR...` from the top menu and open the `analyzePeptidesUI` config dialog. In the dialog: (a) ensure the Logo Summary Table viewer toggle is ON (drives `showLogoSummaryTable: true` → `addLogoSummaryTable()` at `model.ts#L1195`); (b) ensure the Sequence Space toggle is ON (drives `addSequenceSpace` at `model.ts#L1438`); (c) ensure the Clusters mode is set to `MCL` (drives the MCL clustering algorithm on the sequence-space embedding output). Click OK; allow `startAnalysis` to complete — `addSequenceSpace` may take several seconds for the dim-reduction + MCL passes (per atlas description: "runs sequence-space dim-reduction (UMAP / t-SNE per `SequenceSpaceParams`), optionally with DBSCAN clustering, and docks a scatter plot"). After completion, the analysis view contains the Sequence Space scatter plot, the Logo Summary Table viewer, the Cluster Max Activity viewer (if enabled by default), and a `clusters` column appended to the DataFrame.
2. Confirm the clusters column is non-trivial — read the DataFrame's columns via the active TableView and confirm one column has the cluster-id semtype (per `model.addSequenceSpace`'s emit; the column carries integer cluster ids ≥ 0, where -1 is the DBSCAN noise marker). Confirm via DataFrame inspection that ≥ 2 distinct cluster ids exist in the column (single-cluster output would not exercise the LST's per-cluster row enumeration). The presence of multiple clusters is the prerequisite that `calculateClusterStatistics` will emit one stats entry per cluster.
3. Read the Logo Summary Table viewer's grid via `PeptidesModel.getInstance(table).findViewer(VIEWER_TYPE.LOGO_SUMMARY_TABLE)` (per `model.ts#L1187`). Confirm: (a) the viewer is non-null; (b) the underlying DataFrame (`viewer.dataFrame` or the LST's internal `clustersStats` table) contains ≥ 2 rows (one per cluster from Step 2); (c) each row's stat columns (`Members`, `Mean difference`, `p-value` per atlas description: "per-cluster summary grid: WebLogo, activity-distribution histogram, members count, mean difference, p-value") contain non-null numeric values. This is the UI-side assertion that `calculateClusterStatistics` (atlas `peptides.compute.calculate-cluster-statistics`) populated the LST's per-cluster summary correctly. The presence of non-null `Mean difference` and `p-value` values per cluster row confirms the t-test-against-rest stat compute path ran for each cluster.
4. Confirm the per-row stat values are derived (not hard-coded) by sampling row values: the `Members` column's sum across all rows should equal the DataFrame's row count (every row of the source DataFrame belongs to exactly one cluster, including DBSCAN noise cluster -1 if present). The `Mean difference` column's values should span a non-trivial range (max - min > 0), not all zero. The `p-value` column's values should all be in the half-open interval [0, 1] (per the t-test contract). These three sanity assertions on the LST grid confirm `calculateClusterStatistics`'s output shape end-to-end.
5. Click on a row in the LST grid to select a cluster (atlas description for LST: "per-cluster summary grid"; clicking a row drives a selection event that propagates through `model.fireBitsetChanged` per atlas `peptides.model.fire-bitset-changed` at `src/model.ts#L964`). The selected cluster's row indices on the source DataFrame should be highlighted in the underlying grid. Open the Distribution property-panel accordion via the context panel (atlas `peptides.widgets.distribution` at `src/widgets/distribution.ts#L44`: "activity distribution panel with histograms; supports breakdown by monomers, positions, or clusters via `getDistributionForMonomers / Positions / Clusters`"). The accordion's summary record per cluster — per atlas `peptides.compute.get-summary-stats`'s description "collapses a stats item into a summary record (mean, p-value, count, ratio) for the property-panel accordion" — should display non-null `mean`, `p-value`, `count`, and `ratio` values for the selected cluster. This is the UI-side assertion that `getSummaryStats` (atlas `peptides.compute.get-summary-stats`) is invoked correctly with the selected cluster's stats item.
6. Switch the selection to a different cluster (click a different LST row). The Distribution accordion's summary record should update with the new cluster's `mean` / `p-value` / `count` / `ratio` values. Confirm the values differ from the prior selection's values (not stale from the previous selection). This second-selection assertion confirms `getSummaryStats` is re-invoked on selection change, not cached statically.
7. Confirm no null-receiver, division-by-zero, or NaN-in-stats console errors fired during the SAR launch (steps 1-2) or during the LST row inspection + selection (steps 3-6). The combined assertion: (a) the LST grid's rows have non-trivial stats per Step 3-4, AND (b) the Distribution accordion's summary record updates correctly on cluster selection per Step 5-6, AND (c) no console errors fired throughout.

Expected (assertion summary):
- After SAR launch with sequence-space + MCL enabled, the DataFrame has a clusters column with ≥ 2 distinct cluster ids.
- The Logo Summary Table viewer's grid contains ≥ 2 rows (one per cluster), each with non-null `Members`, `Mean difference`, and `p-value` values — confirming `calculateClusterStatistics` populated the per-cluster summary.
- LST row stat values pass sanity checks: `Members` sums to the source DataFrame's row count; `Mean difference` spans a non-trivial range; `p-value` values are in [0, 1].
- Clicking an LST row selects the cluster and drives the Distribution property-panel accordion's summary record to show non-null `mean` / `p-value` / `count` / `ratio` — confirming `getSummaryStats` is invoked on the selected cluster's stats item.
- Switching selection to a different LST row updates the Distribution accordion's summary record with the new cluster's values (not stale) — confirming `getSummaryStats` is re-invoked on selection change.
- No null-receiver, division-by-zero, or NaN-in-stats console errors fire throughout the LST + Distribution accordion chain.
