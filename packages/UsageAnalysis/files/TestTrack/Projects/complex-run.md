## Summary

**Total scenario run (with model): ~35m.** 17 of 22 fine-grained sub-steps PASSED in the
MCP run (essentially everything that has a JS API path). 5 sub-steps SKIP because the
dev environment lacks the prerequisite (no Space-bound tables; package-bound
queries/scripts not movable; drag-and-drop and second-user re-auth not scriptable). 1
sub-step FAILS with a real platform issue: **`permissions.grant()` rejects the project
author with "You don't have a permission to share this object"** — reproducible on dev
as `admin`. The Playwright spec covers the testable flow end-to-end and assertively
fails on the share regression and on the secondary "Save a copy from a reopened project
didn't list-refresh in time" flake (Steps 7-8/9), so the spec is useful as a real
regression detector rather than a green-by-construction smoke test.

## Retrospective

### What worked well

- **Multi-source ingestion** is fluid via JS API: `dapi.files.readCsv` for file shares,
  `functions.call('Samples:PostgresProducts')` for DB/query results, and direct call
  for scripts. All four "open from..." cases worked.
- **`groupBy().agg().aggregate()`** is a clean substitute for the Pivot/Aggregate UI
  push-to-workspace flow.
- **`grok.data.joinTables(t1, t2, k1, k2, c1, c2, type)`** does what the `Data > Join
  Tables` dialog does — including over a DB-result + Query-result mix.
- **Project save → reopen** preserves the table set, schema, and Data Sync source for
  `db_products` (rowCount matches a fresh query call). The renamed names persist for
  non-sync tables.
- **Renaming entities via JS API** (`entity.friendlyName = '...'; dapi.<type>.save(entity)`)
  worked for Project, Query, and Script. The slug `name` correctly stays on the original
  value (only `friendlyName` is mutable post-save).
- **`Save original project`** mode IS the right entry-point for "update an existing
  project in place" — see `What did not work` for what it doesn't cover.

### What did not work

- **`grok.dapi.permissions.grant(proj, user_or_group, edit)`** server-rejects with
  *"You don't have a permission to share this object"* even when the caller is admin
  AND the project's author. Reproducible across `User`, `user.group`, and fully-fetched
  `Group` arg shapes. Blocks Step 12 entirely.
- **`Save original project` does NOT add newly-opened tables to an open project.** Flow:
  open project (8 tables) → open 3 more tables → Save dialog → "Save original project" →
  OK → reopen → only the original 8 tables come back. The 3 added tables are dropped.
- **`grok.data.db.query(connId | name | "DS:Name", sql)`** fails three different ways:
  passing a UUID → `"Unable to parse string"`; passing the bare name → `(intermediate
  value).query is not a function`; passing `Postgres:Chembl` → `"Unable to get project
  asset 'Chembl'"`. Same connection works fine when wrapped in a published query
  (`Samples:PostgresProducts`), so the platform routing works; the public DB-query API
  is broken.
- **`dapi.queries.filter('name = "X"').first()` returns null even when the query exists**
  (already documented in `uploading-run.md`). Same pattern affects scripts
  (`scripts.filter('name = "RowsTest"').first()` returned null on first try). The
  workaround is `await dapi.<type>.list({pageSize: 500})` + JS `.find()`.
- **`grok.dapi.projects.list({pageSize: N})` (no filter) returns only Dashboards** —
  Spaces (`isDashboard = false`) require an explicit `.filter('isDashboard = false')`
  to surface. Easy to miss.
- **Move-to-Space JS API does not exist.** `proj.parent = rulesSpace` sets the in-memory
  reference and `dapi.projects.save(proj)` returns successfully (44s of round-trip), but
  re-fetching shows `parent = null`. Same with `space.addChild(proj); save(space)`.
- **Save dialog: Data sync toggle visibility is per-table source binding.** Of 8 tables,
  only `db_products` (the one with `.DataQuery.id` tag) showed the toggle; 7 others had
  `style="display:none"`. After reopen in a fresh browser context, even `db_products`
  may show the toggle as OFF rather than restoring the saved ON state — environment-
  dependent timing.
- **`Save a copy` from a just-saved project may rename the existing project rather than
  cloning.** Observed in MCP run: saved Complex_DS2 → opened it → Save a copy as
  Complex_RenamedDS → ID equals the just-saved Complex_DS2's ID, and Complex_DS2 is gone
  from the list. Likely a session-state issue where the in-shell project pointer is
  treated as the target of the "copy" save.
- **`db_products` reverts to the original column tags / table name on reopen** because
  Data Sync overwrites the per-table name with the source query's default. Renaming
  `db_products` → `db_products_r` is preserved on disk but not on the next open.
- **Drag-and-drop into Browse > Dashboards** is not Playwright-scriptable for the
  required gesture sequence. SKIP for Step 4.
- **Second-user re-auth** has no JS API — the Playwright spec would need a second token
  injection through `loginAsSecondUser(page)`, which requires `DATAGROK_AUTH_TOKEN_2`.

### Suggestions for the platform

1. **Fix `permissions.grant()` for project authors.** Returning "You don't have a
   permission to share this object" to admin (also the author) is wrong; either the
   server-side ACL check is bugged, or the JS API is sending an arg shape the server
   rejects. Expose a clearer error if it's the latter.
2. **Surface a `Project.addTable(t)` / `Project.tables` mutator.** The current "Save
   original project" silently dropping new tables is surprising; an explicit
   `proj.tables.push(t); await dapi.projects.save(proj)` would let scripts and tests do
   the equivalent of drag-and-drop into Dashboards.
3. **Fix `grok.data.db.query()`** to accept either a connection ID or a name — currently
   neither shape works on dev. Either form should be a one-liner.
4. **Make `dapi.<type>.filter('name = "X"').first()` either work or error**, not silently
   return null / unrelated rows. Same finding as the prior `uploading-run.md` retro.
5. **Expose `Project.moveTo(parent)`** for moving projects into Spaces or file shares
   programmatically. Currently only the UI drag-and-drop path works.
6. **Persist the `friendlyName` of source-bound tables across reopen.** Right now Data
   Sync reverts the saved table name to the source query's default, contradicting the
   user's explicit rename.
7. **`Save a copy` should always create a new project entity**, never renaming the
   project that's currently in shell — this is the user's expectation when picking the
   "copy" radio.
8. **`grok.dapi.projects.list()` should include Spaces by default** (or document the
   `isDashboard = false` filter prominently), since "list all projects" is a common
   sanity-check call.
9. **Add `data-testid` (or stable `name=`) to the per-table Link/Clone/Data sync
   controls in the Save dialog**, scoped to a per-table row. The current scoping
   (eight `[name="input-host-Data-sync"]` siblings, only one visible) is brittle.

### Suggestions for the scenario

1. **Drop `Table from a Space`** unless the dev/test environment is seeded with at least
   one non-empty Space. Otherwise this whole sub-step is unrunnable.
2. **Either replace `Move to file share / Space`** with a JS-API-driven move, or rewrite
   the step to use a UI gesture that's actually scriptable (e.g. right-click → Move to
   ▸ pick from a combo). Currently "drag-and-drop into the Browse tree" is the only
   path and it's not test-automatable.
3. **Add a setup section** with required prerequisites: a non-admin user account, a
   second-user dev key (or a hint on how to set `DATAGROK_AUTH_TOKEN_2`), at least one
   Space, and a user-owned (not package-bound) Query and Script. Without these,
   roughly 30% of the scenario sub-steps SKIP.
4. **Spell out which Save mode is required at each step.** The current scenario says
   "Save … as a project with Data Sync enabled" — but the Save dialog has three radio
   modes with different semantics, and "Save a copy" is what most steps actually want.
5. **Make the rename step explicit about which entities to rename and to what.** The
   current "rename all tables inside it" is ambiguous about whether `db_products`
   (Data-Sync-bound) is included; spelling out "rename only non-sync tables" or
   "include sync tables and accept that the name will revert on reopen" would avoid
   the gotcha.
6. **Specify which file share is the move target.** "Move to any file share" is
   ambiguous when the user has dozens of Apitests-* shares.
7. **Note the dialog cascade.** Steps 5-7 each open the Save dialog on top of the
   previous result; the resulting in-shell project pointer is non-trivial. Either
   explicitly `closeAll()` between steps, or call out the in-shell-pointer caveat.
