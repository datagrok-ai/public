## Unresolved ambiguities

- **Setup step "Set the Hierarchy to: CONTROL, SEX, and RACE":** the original scenario does not specify the UI action path. Two plausible UI paths exist: (a) drag the columns from the column-bag onto the Tree viewer's hierarchy slot; (b) open the Tree viewer property panel (Gear icon) and edit the `hierarchyColumnNames` property via the column-multi-selector. The existing `tree-spec.ts:37` chooses neither UI path — it uses `tree.setOptions({hierarchyColumnNames: ['CONTROL', 'SEX', 'RACE']})` (JS API path) deterministically. Migrator preserved the original wording verbatim; Automator may continue to use the JS API setOptions path (deterministic) or, if a follow-up specifies UI driving, switch to property-panel editor. Flagged at chain level as unresolved ambiguity #3 (chain YAML lines 398-406, priority: low). Preference per chain: JS API path (`viewer.setOptions`) for determinism. **Cross-reference:** chain `unresolved_ambiguities` entry for tree.md retains this for downstream visibility.
- **"Filtered count" semantics:** the original Expected results state "Filtered count = 0", "= 2", "= 176" without specifying which counter is asserted (the Filter Panel's visible row count, the table view's row indicator, the Tree viewer's branch-aggregate count, or the intersection of selection ∧ filter bitsets). The existing `tree-spec.ts` interprets the count as the **overlap** between `df.selection` and `df.filter` bitsets (not the filter-only count). This migration preserves "Filtered count" wording verbatim and uses the spec's overlap interpretation; the count values 0, 2, 176 are consistent with the overlap reading per the spec assertions. Flag for QA pair review if downstream Gate B fails on count semantics.
