## Extractors (12), in build order

1. **`library`** — walks `libraries/*/`. Emits `Library`, `LibraryModule`, `DependsOn`, `ImportsFromModule`.
2. **`ts_plugin_package`** — walks `packages/*/`. `Package`, `RegisteredFunction` (from `package.g.ts` annotations — fallback to `package.ts`/`package.js`), `PackageProperty`, `Exports`, `HasProperty`, `DependsOn`.
3. **`ts_plugin_scripts`** — `scripts/*.{py,r,js,jl,m}` headers → `Script` + `HasScript` + `RequiresEnvironment`.
4. **`ts_plugin_queries`** — `queries/*.sql` blocks → `DataQuery` + `HasQuery` + `UsesConnection`.
5. **`ts_plugin_connections`** — `connections/*.json` → `DataConnection` + `HasConnection`.
6. **`ts_plugin_environments`** — `environments/*.yaml` → `ScriptEnvironment` + `HasEnvironment`.
7. **`ts_plugin_dockerfiles`** — `dockerfiles/*/Dockerfile` → `DockerContainer` + `HasContainer`.
8. **`ts_plugin_changelog`** — `CHANGELOG.md` → `ChangelogEntry` + `HasChangelogEntry` + `JiraTicket` + `MentionsTicket`. Handles bullets AND paragraph-style entries.
9. **`docs`** — `help/**/*.md` → `DocPage` + `HelpAnchor` + `LinksTo` + `HasAnchor`. Plus tutorial parser (`Tutorial` + `TutorialTrack`). Plus `//help-url:` annotation → `Documents` (only ~4 in `.g.ts`, ~12 in scripts, ~21 viewer assignments — most are LLM-derived in `doc_linking`).
10. **`cross_package`** — scans every `packages/*/src/**/*.ts/.js` for `grok.functions.eval/call/find('Pkg:fn')`, `_package.functions.*`, `DG.Func.byName/find` → `Calls` edges. **Zero `IMPORTS_FROM_PACKAGE`** (Datagrok deliberately avoids cross-pkg TS imports for webpack bundle reasons).
11. **`files`** — walks every source dir under packages + libraries. Emits `File` + `ContainsFile` + `DefinedIn` (from existing entities' `paths[]`). 4595 File nodes.
12. **`semtypes`** — scans `RegisteredFunction.inputs/outputs/meta` for `semType` references → `SemanticType` + `Detects/Consumes/ProducesSemtype`.
13. **`js_api`** — `js-api/src/`. `TsClass(is_jsapi=True)`, `TsMethod`, `TsEnum`, `TsConstant`, `TsInterface` plus `JsApiNamespace`, `DapiEndpoint`, `JsEventStream`, `GeneratedBinding`, `UiComponent`. Plus `HasMethod`/`InNamespace`/`ExtendsClass`/`DelegatesTo`/`TypedFor`/`DefinesSemtype`.
14. **`ts_code_structure`** — same regexes as `js_api` but on `{packages,libraries}/*/src/`. `TsClass(is_jsapi=False)` etc. + `HasClass` + `HasFunction` + `HasMethod`. **All 1170 TsClass / 5097 TsMethod / 2550 TsFunction / 1243 TsInterface across packages and libraries.**
15. **`js_api_usage`** — scans plugin TS for `import * as DG/grok/ui from 'datagrok-api/...'` + `DG.<X>` / `grok.dapi.<x>` / `grok.events.<x>` / `ui.<x>` references → `ImportsNamespace`/`UsesApiClass`/`UsesApiEnum`/`CallsDapiEndpoint`/`SubscribesToEvent`/`UsesUiComponent`. Resolves only against `is_jsapi=True` nodes.
16. **`feature_files`** *(POST-PASS)* — derives `IsImplementedIn`/`IsTestedIn` from `Feature -[:PART_OF_FEATURE]- member -[:DEFINED_IN]-> File`. **THIS IS CURRENTLY TOO SHALLOW** — see Open Issues below.
17. **`feature_classify`** *(POST-PASS)* — derives `Feature.is_user_facing` (bool) and `Feature.interaction_kind` from PART_OF_FEATURE member roles. **MAY NOT HAVE RUN PROPERLY** — confirm by `MATCH (f:Feature) WHERE f.interaction_kind IS NULL RETURN count(f);`. If most are NULL, re-run `py build.py --extractors feature_classify`.

## Enrichers (LLM-driven, prepare/apply)

User has Claude Max — **route enricher work through the Agent tool, not Anthropic SDK** (no API key). The contract:

1. `prepare()` reads JSONL, writes per-unit input JSON to `data/enrichment/<name>/inputs/<unit>.json` plus a shared `PROMPT.md`.
2. **The orchestrator (this Claude Code session) launches one Agent per input file.** Each Agent reads its input + the prompt, writes the output JSON.
3. `apply()` validates outputs, merges into canonical JSONL.
4. `build.py` rebuilds Kuzu.

**Currently registered enrichers:**

- **`feature_clustering`** — per-package. 79 packages → 458 Feature nodes, 2810 PartOfFeature edges. Run with `py enrich.py prepare --enricher feature_clustering` then dispatch agents per package, then `py enrich.py apply --enricher feature_clustering`.
- **`doc_linking`** — per-`/help/` subtree. ~660 `DOCUMENTS` edges + 220 `PART_OF_FEATURE` edges. Same lifecycle. Apply step ALSO seeds `PART_OF_FEATURE` edges when target is a Feature.

## Hard-won lessons (do not repeat the mistakes)

1. **`build.py --clean` USED TO wipe `data/` entirely**, including `data/enrichment/` (the LLM agent outputs). Re-running 90+ agents costs hours. **Fixed:** `--clean` now only wipes `data/nodes/` and `data/edges/`. NEVER add `data/enrichment/` back to that list. NEVER auto-wipe enrichment outputs from any future tool. Treat them as immutable artifacts.

2. **Rate limits hit when launching ~79 agents in one go.** Anthropic returns "Server is temporarily limiting requests (not your usage limit) · Rate limited" on a chunk of them after the first wave. **Mitigation:** launch waves of 25–30, then re-launch the failed ones after 5 min. Some agents simply stall — re-launch those too.

3. **Some agents take 5–10 min** for big inputs (Chem at 153 functions × 31 scripts × 25 docs took 10 min). One stalled at 600s. Plan accordingly.

4. **`JsApi*` was renamed to `Ts*`** (TsClass, TsMethod, TsEnum, TsConstant, TsInterface) to be generic across layers. `is_jsapi: bool` distinguishes. The 5 JS-API-only kinds (`JsApiNamespace`, `DapiEndpoint`, `JsEventStream`, `GeneratedBinding`, `UiComponent`) kept their JsApi prefix.

5. **The user wants generic concepts (classes, interfaces, functions) to apply to packages and libraries too**, not just JS API. That's why we have `ts_code_structure.py` alongside `js_api.py`.

6. **Cross-package TS imports literally don't exist** in this codebase. Datagrok packages route all cross-package interaction through `grok.functions.eval/call/find('Pkg:fn')` or `DG.Func.byName/find` to avoid webpack bundle bloat. So `ImportsFromPackage` always has 0 edges. Don't try to extract them.

7. **`grok.functions.eval('Pkg:fn')` doesn't always resolve.** ~55 of 114 detected calls land on a `RegisteredFunction`. The other 59 reference function names that don't exist as `RegisteredFunction.name` — they probably need a `friendly_name` fallback resolver. **TODO.**

8. **Edges with non-existent endpoints are silently skipped during Kuzu COPY** with a count message. Watch for those messages — many are legit (e.g. `pkg-npm:` references resolved post-hoc), some are bugs.

9. **Empty lists become NULL in Kuzu.** `_coerce` in `build.py` collapses `[]` to NULL because Kuzu's parameterized binding can't infer element type for empty lists. So query with `WHERE x IS NOT NULL` before `size(x)`.

10. **`default` is a reserved Cypher word.** We renamed `PackageProperty.default` to `default_value`. If you add a field, avoid: `default`, `match`, `from`, `to`, `value`, `key`.

11. **Kuzu file/dir handling differs by version.** `--clean` checks `is_dir()` before choosing `rmtree` vs `unlink`. Don't break this.

12. **The viewer (`kg.html`) starts empty by design.** User clicks node-kind toggles in the sidebar legend or picks a Package. Rationale: 27K nodes blow up any layout.

## Open issues (in priority order)

### 1. ✅ `IS_IMPLEMENTED_IN` / `IS_TESTED_IN` shallowness — RESOLVED 2026-05-10

Both halves of the fix shipped:

**Deterministic (`extractors/feature_implementation.py`):**
Parses each package's `src/package.ts` for `@grok.decorators.<role>({name: 'X'})`-annotated `static` methods (and the older `//name:` + `export function` pattern). For each method body, walks the import chain ~2 hops and emits `IS_IMPLEMENTED_IN(feature, file, derived_by=AST)` for every in-package file referenced. Tests scanned by name/method-name match. Skips `package.g.ts`/`package-api.ts`.

**LLM enricher (`enrichers/feature_implementation.py`):**
Per-package input JSON contains: package metadata, full src/test file listings, feature descriptions, member IDs, and the deterministic `ast_baseline` per feature. Agent uses Read/Glob/Grep to refine. Apply step writes `derived_by=LLM` edges with confidence per file.

**Final coverage:**
- IS_IMPLEMENTED_IN: 3358 edges (1362 AST + 1111 filesystem + 885 LLM)
- IS_TESTED_IN: 518 edges (150 AST + 110 filesystem + 258 LLM)
- All 79 packages dispatched (in 3 waves of ~26 each), 0 validation problems.

**Bio atomic-level structure** now correctly resolves to 16 files including
`widgets/to-atomic-level-widget.ts`, `utils/sequence-to-mol.ts`, and the full
`utils/helm-to-molfile/converter/*` pipeline — the case the user flagged.

### 2. **`feature_classify` may not have run** in the final rebuild

Verify with: `MATCH (f:Feature) WHERE f.interaction_kind IS NULL RETURN count(f);`. If most are NULL, just re-run `py build.py --extractors feature_classify`. The extractor reads existing `data/nodes/Feature.jsonl` and overwrites it with `is_user_facing` + `interaction_kind` populated.

### 3. **CALLS resolution is 55/114** — friendly-name fallback would recover most of the rest

In `cross_package.py`, when `grok.functions.call('Pkg:foo')` doesn't resolve to `func:Pkg:foo`, also try matching by `RegisteredFunction.friendly_name`. Quick fix.

### 4. **`DEFINES_SEMTYPE` 27 skipped, `LINKS_TO` 1623 skipped, `USES_CONNECTION` 168 skipped**

These are real findings (broken doc links, semtype names not in detector pool, cross-pkg connection refs). Don't "fix" them by relaxing constraints — they should surface as gaps.

### 5. **Doc layer is sparse for `client-facing` distinction**

Currently `Feature.is_user_facing` exists (when feature_classify runs). But "audience" (chemist vs developer vs admin) is only on DocPage, not Feature. Could be propagated.

### 6. **No commits/git layer**

Schema has `Commit` defined. Extractor never written. Would let us answer "which features were touched in last week's commits?"

### 7. **No ApiTest / PlaywrightScenario nodes**

Schema entities defined. Extractors never written. Without these, `Covers` edges are empty, and we can't derive proper "which test exercises this feature" beyond filename heuristic.

## Tools & key commands (cheat sheet)

```powershell
# Build the graph from canonical JSONL
py build.py                                 # full rebuild (preserves enrichment/)
py build.py --extractors feature_classify   # one extractor only

# Query

py qq.py --stop                                       # release DB lock for build
py qq.py --restart                                    # after rebuild
py qq.py --json "..."                                 # machine-readable
py query.py --demos                                   # one-off queries (no server)

# Tests + viz

```

## What the user values (do not lose this perspective)

- **Quality over speed.** They flagged shallow `IS_IMPLEMENTED_IN` because it's misleading. Better to have fewer, accurate edges than many, shallow ones.
- **Correctness over coverage** in extraction. Fix the deterministic side before piling on LLM.
- **Hybrid extraction is the model.** Deterministic for what we can prove, LLM agents for what we have to infer. Always with provenance.
- **The graph is meant to be the entry point** for users, devs, and future agents (per `CLAUDE.md`). It should be trustworthy. Misleading edges are worse than missing edges.
- **No destructive ops without permission.** `--clean` taught us this.
