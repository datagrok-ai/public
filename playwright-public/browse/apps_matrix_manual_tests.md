# Browse — Apps matrix (manual test cases)

Мануальные тест-кейсы для обхода **всех приложений** в `Browse > Apps`. Каждое приложение
открывается **кликом по узлу дерева Browse (по названию)**; приложения, разворачивающиеся в
поддерево с данными, обходятся вглубь — каждый под-узел открывает свой view, с которым
выполняется минимальное взаимодействие. Главная проверка — **отсутствие любых ошибок** (консоль,
балуны, Context Panel).

Автоматизация: `apps_matrix.test.ts` (этот же каталог). Структура кейсов 1:1 с массивом `APPS` и
логикой обхода в коде.

---

## 1. Окружение и общие предусловия

- **Инстанс:** `https://dev.datagrok.ai/` под залогиненным пользователем с правами на демо/данные.
- **Источник матрицы:** живое дерево `Apps`, снятое с dev (см. `browse_tests_context.md`).
- **Открыто:** боковая панель **Browse**, узел **Apps** развёрнут.
- Навигация строго **по названию** узла (стабильный атрибут `name="tree-Apps---…"`).

## 2. Скоуп

Покрываются **все верхнеуровневые узлы под `Apps`, КРОМЕ**:
- `Demo` — покрыт `demo_apps.test.ts`;
- `Compute > Model Hub` — покрыт `modelhub.test.ts`;
- `Alation` — приложение снято с поддержки (осталось только на dev), исключено из матрицы.

## 3. Как открывается приложение (поведение платформы)

- Клик по листу-приложению вызывает его `@app`-функцию → открывается новый view (имя view = имя
  узла), Browse-панель может смениться на Toolbox.
- Групповые приложения строят своё поддерево **лениво** (`@appTreeBrowser`, иногда обращается к БД
  или docker-контейнеру) — узлы-дети появляются с задержкой и **постепенно** (streaming).
- Часть приложений зависит от **docker-контейнера** (MolTrack, Admetica, Retrosynthesis,
  Docking/Boltz через `bio`, Preclinical Case) — если контейнер не поднят, view не функционирует.
- Часть приложений — **внешние коннекторы** (Benchling, CDD Vault, Revvity Signals, Chemspace) —
  зависят от стороннего SaaS, доступного только на dev.

## 4. Глубина покрытия («sample»)

Чтобы suite укладывался в таймаут CI (1 worker) и не взрывался комбинаторно (один
`Preclinical Case` = 216 узлов):

- **Лист-приложение** (`Flow`, `PopPK`, `Tutorials`) → открыть, повзаимодействовать, проверить.
- **Групповое приложение** → открыть **ВСЕ** верхнеуровневые дочерние узлы (тулы / инстанции), и
  для каждого раскрывающегося ребёнка пройти **ОДНУ репрезентативную цепочку** (первый ребёнок на
  каждом уровне, до листа). Так одна дата-инстанция открывается «вглубь» без обхода всех
  однотипных дата-листьев.
- `Tutorials` — только открыть (`runOnly`); автотест прохождения туториалов — отдельно, позже.

## 5. Общий тест-кейс (для каждого узла)

**ID:** `Browse-AppsMatrix-<App>[-<под-узлы>]`
**Тип:** smoke.

**Шаги (действие → ожидаемая реакция):**
1. Раскрыть `Apps` → появляются приложения.
2. Раскрыть приложение (по названию) → появляются его дочерние узлы (дождаться достройки дерева).
3. Кликнуть узел → открывается его view.
4. **Поклацать во вью:** клик по телу view; если есть grid — клик по ячейке; навести курсор на
   основной viewer.
5. Открыть **Context Panel** (F4) + **Expand all** → все панели свойств отрисовываются.

**Итоговая проверка (для всех узлов):**
- ✅ нет uncaught JS-ошибок (`pageerror`);
- ✅ нет `console.error`;
- ✅ нет error-балунов;
- ✅ при раскрытии Context Panel ни одна панель не падает.

## 7. Обработка ошибок (что НЕ считается багом)

Suite честно краснеет на реальных дефектах Browse/UI, но **толерирует** заведомо не-дефекты —
по узлу, на котором ошибка возникла:

1. **Инфраструктура (всегда):** обрывы соединения, `Container … not started`, сырые HTTP-ответы
   бэкенда `4xx/5xx`. Сигнатуры узкие, чтобы не замаскировать реальный баг с числом в тексте.
2. **Docker-зависимые апы:** при docker-ошибке проверяется **живой статус контейнера**
   (`grok.dapi.docker.dockerContainers`). Контейнер `started` + ошибка = **реальный баг (fail)**;
   контейнер не `started` (`starting`/`error`/`stopped`/отсутствует) = инфра, **толерируется**.
3. **Коннекторы** (Benchling, CDD Vault, Revvity, Chemspace): зависят от внешнего SaaS, доступного
   только на dev → все их ошибки толерируются.
4. **Шум от взаимодействия:** `ResizeObserver: parameter 1 is not of type 'Element'` и React
   error-boundary эхо — провоцируются нашим `pokeView` по монтирующемуся React-редактору; при
   спокойном открытии в изоляции их нет → не дефект навигации.
5. **Async-bleed:** docker-down / коннектор-сабапа фонит балунами, которые «налипают» на соседний
   узел → после выхода из такой сабапы делается `closeAll`, чтобы её заглушить.

Каждая толерация логируется строкой `ⓘ <узел>: N error(s) tolerated — <причина>`.

## 8. Портируемость (другие стенды / CI)

Матрица снята на dev (полный набор). На минимальном стенде пакет приложения не установлен →
узел приложения отсутствует → весь app `test.skip()`-ается; отсутствующие под-узлы пропускаются
индивидуально. Жёстко зашитого allowlist нет — один и тот же файл корректен и на dev, и на
урезанном стенде. Защита от false-green: если приложение присутствует (группа), но не открылся ни
один узел — это провал загрузки поддерева (fail), а не «успех вхолостую».

## 9. Результаты прогонов на dev (триаж)

Полный прогон всех 11 тестов на `dev.datagrok.ai` (2026-06-23): **11 passed / 0 failed**.

- 🐞 **GROK-20254** — Hit Triage / Hit Design бросали повторяющийся
  `Cannot read properties of null (reading 'id')` по таймеру, пока вью открыт. Найдено этим
  suite, подтверждено в изоляции, заведён тикет; на момент финального прогона **уже исправлено на
  dev** (тест зелёный).
- 🔧 **Plates / Search plates** — `RangeSlider.setValues` падал при `grid.columns.add` — оказался
  **гонкой теста** (`closeAll` уничтожал вью, пока его debounced `search().then()` ещё летел). Не
  баг платформы. Лечится: не делать `closeAll` между под-узлами + ждать `networkidle`+settle.
- 🔧 **ResizeObserver `<Editor>`** (MolTrack Register / Reaction Enumerator) — провоцировался
  `pokeView`; в изоляции чисто → шум, занесён в игнор.
- ⓘ **MolTrack** — `Container not started` (контейнер `moltrack` = `starting`): инфра,
  толерируется по живому статусу docker.

## 10. Заметки

- Список приложений снят автоматически; при изменении набора на сервере перегенерировать матрицу
  (см. `browse_tests_context.md`).
- `heavy` — приложение запускает серверные/docker-вычисления, открывается дольше (увеличенный
  таймаут теста).
- Глубокие однотипные дата-листы (датасеты домена клинического/преклинического study) **не**
  обходятся целиком — это и есть «sample».
