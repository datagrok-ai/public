# Datagrok Browse — мануальные тест-кейсы

**Инстанс:** https://dev.datagrok.ai/
**Объект тестирования:** Browse (навигация, дерево, view-режимы), все секции дерева
**Тип:** UI / мануальные (заготовка под перенос в Playwright)
**Источники:** docs `datagrok.ai/help/datagrok/navigation/`; Jira GROK (Test Track, баги Browse)

---

## Соглашения

- **ID** кейса: `Browse-<Область>-<NN>` — формат совместим с Test Track (ср. `Scripts-Edit-6`).
- **Приоритет:** P1 (критичный путь) … P3 (edge / косметика).
- **Тип:** `smoke` (базовая работоспособность), `functional` (проверка поведения), `regression` (привязан к конкретному багу), `negative` (ошибочный/пустой ввод, отсутствие прав).
- Каждый кейс: **Цель** → **Тип** → **Предусловия** → **Шаги** (действие + ожидаемая реакция UI) → **Итоговая проверка** → **Постусловия/очистка**.
- Шаги атомарные: одно действие на строку с ожидаемым результатом. При автоматизации действие → `await`-вызов, ожидание → `expect`. Предусловия → `beforeEach`/фикстура; «Постусловия» → `afterEach`/teardown.
- Критерии измеримые: где возможно, указаны конкретные значения (тайминги, наборы пунктов меню, точное состояние), а не «корректно/релевантно».
- Селекторы пока не зафиксированы (снимем при первом прогоне на dev, см. «Заметки для автоматизации»). В тексте элементы названы по UI-подписи/роли.
- `ref:` — связанный баг/тикет в Jira.

**Глобальные предусловия (если в кейсе не сказано иного):**
- Пользователь залогинен на https://dev.datagrok.ai/ под учёткой с правами на демо-данные; открыта стартовая страница.
- Включён перехват console-ошибок (см. ниже).
- **Context Panel открыта** во всех Browse-тестах (тоггл F4). Если в процессе теста панель закрылась — переоткрыть. Для секции 18 («узлы дерева без ошибок») Context Panel должна быть **полностью развёрнута** (Expand all info panes). Это требование **не применяется** к будущим matrix-наборам по Demo / Apps.

**Тестовые учётки (см. `playwright-tests/.env`):**
- `DATAGROK_LOGIN` = `opavlenko+playwright@datagrok.ai` — основная, **admin** (используется по умолчанию во всех кейсах).
- `DATAGROK_SHARING_LOGIN` = `opavlenko+pwsharing@datagrok.ai` — вторая dev-учётка, **без admin-прав**. Используется в кейсах с шарингом и проверкой ограничений по правам (Tree-07, Platform-03, MyStuff-04 и т.п.). Для таких кейсов готовим отдельный `storageState`.

**Что считаем ошибкой (для проверок «без ошибок»):**
- `pageerror` (необработанное исключение страницы);
- `console.error` в консоли браузера;
- datagrok-балун с текстом ошибки (red/error notification);
- стек Dart/JS, всплывший в UI или Context Panel.
Любого из этого достаточно, чтобы кейс считался проваленным.

**Тестовые данные:** на dev присутствуют демо-файлы (Files > Demo), демо-БД (Databases) и демо-дашборды (Dashboards). Кейсы опираются на них. Для создаваемых в тесте сущностей использовать уникальный префикс имени (например `qa_autotest_<timestamp>`), чтобы упростить очистку.

### Browse-Nav-01 — Открытие панели Browse с Sidebar (P1)
**Цель:** Browse открывается и содержит обязательные секции дерева.
**Тип:** smoke
**Предусловия:** глобальные.

**Селекторы:** `SIDEBAR_BROWSE_ICON`, `BROWSE_PANEL_TAB_PANE`, `BROWSE_HEADER`, `treeGroupByName('My stuff' | 'Spaces' | 'Apps' | 'Files' | 'Databases' | 'Platform')`, `treeItemByName('Dashboards')` — см. `selectors.md` / `selectors.ts`.

## 2. Browse tree — работа с узлами

### Browse-Tree-01 — Раскрытие и сворачивание узла (P1)
**Цель:** expand / collapse узла по клику на стрелку.
**Тип:** smoke
**Предусловия:** дерево открыто, секция **Files** свёрнута.
**Шаги:**
1. Кликнуть стрелку раскрытия слева от узла **Files**. → Узел раскрывается, появляются дочерние элементы.
2. Осмотреть дочерние узлы. → Видны My files, Spaces, App Data, Demo; стрелка в состоянии «раскрыто».
3. Кликнуть стрелку сворачивания узла **Files**. → Узел сворачивается, дочерние элементы скрываются.

**Итоговая проверка:** после раскрытия видны дочерние узлы, стрелка «раскрыто»; после сворачивания дочерние узлы отсутствуют в DOM/невидимы, стрелка «свёрнуто».
**Селекторы:** `treeGroupByName('Files')`, `treeExpandArrow('Files')`, `TREE_EXPAND_ARROW_EXPANDED`, `treeNodeChildren(page, 'Files')` — см. `selectors.md` / `selectors.ts`.
**Постусловия/очистка:** не требуется.

### Browse-Tree-02 — Клавиатурная навигация по дереву (P2)
**Цель:** управление деревом с клавиатуры — навигация, expand/collapse, активация.
**Тип:** functional
**Предусловия:** фокус установлен на узле дерева (клик по узлу с детьми, дочерние свёрнуты).
**Шаги:**
1. Нажать **↓**. → Фокус перемещается на следующий видимый узел.
2. Нажать **↑**. → Фокус возвращается на исходный узел.
3. Нажать **→**. → Узел раскрывается, появляются дочерние элементы.
4. Нажать **←**. → Узел сворачивается.
5. На узле-элементе (item, не group) нажать **Enter** (или **Space**). → Элемент открывается в центре (как одиночный клик).

**Итоговая проверка:** ↑↓ переключают выделение (в каждый момент ровно один узел выделен); → раскрывает, ← сворачивает; Enter/Space открывает элемент.
**Селекторы:** `treeNodeContainer(page, name)`, `treeNodeSelected`, `TREE_EXPAND_ARROW_EXPANDED`, клавиши `ArrowUp` / `ArrowDown` / `ArrowLeft` / `ArrowRight` / `Enter` / `Space` — см. `selectors.md` / `selectors.ts`.
**Постусловия/очистка:** не требуется.

## 3. Browsing mode vs Persistent view

### Browse-View-01 — Одиночный (unpinned) vs двойной (persistent) клик (P1)
**Цель:** browsing mode — одиночный клик создаёт unpinned (заменяется), двойной — persistent (не заменяется).
**Тип:** functional
**Предусловия:** в дереве есть три демо-файла A, B, C.
**Шаги:**
1. Одиночный клик по A. → A открывается как unpinned view.
2. Одиночный клик по B. → View A заменяется на view B; число открытых табов не выросло.
3. Двойной клик по C. → C открывается в persistent view; B остаётся открытым.
4. Одиночный клик по A. → A открывается, C остаётся открытым (не вытеснен).

## Заметки для автоматизации (Playwright)

- **Базовый URL:** `https://dev.datagrok.ai/` → в `baseURL` конфига. Авторизацию — через `storageState`/глобальный setup, не повторять логин в каждом тесте.
- **Структура кейса → код:** «Предусловия» = `beforeEach`/фикстура; шаг = действие (`await ...`) + проверка реакции (`expect`); «Итоговая проверка» = финальный `expect`; «Постусловия/очистка» = `afterEach`/teardown.
- **Теги тестов:** использовать поле «Тип» для тегов — `@smoke`, `@functional`, `@regression`, `@negative`. Smoke-набор (все P1 smoke) — для прогона на каждый деплой.
- **Детект ошибок:** глобальный слушатель `page.on('pageerror')` и `page.on('console')` (фильтр по `error`); плюс проверка отсутствия datagrok error-балуна (селектор уточнить). Вынести в общий хелпер `assertNoErrors(page)` и звать в финале каждого «без ошибок»-кейса.
- **Создание данных для предусловий:** объекты для Nav-06, Files-05, MyStuff-04, ModelHub-05 создавать через API/второй контекст в setup, а не руками в UI — так стабильнее и быстрее.
- **Очистка:** все кейсы с созданием/переименованием/share/сохранением чистят за собой (см. «Постусловия»); имена с префиксом `qa_autotest_<timestamp>` + общий teardown, который удаляет всё по префиксу (страховка от упавших тестов).
- **Асинхронность:** дерево и списки грузятся с сервера — ждать появления узлов/элементов, не `sleep`.
- **Тайминги:** для Apps-01 порог 5 c вынести в константу конфига, чтобы легко менять.
- **Селекторы:** реальные значения хранятся в `selectors.md`. На первом прогоне UI-разведки снять стабильные `data-*`/роли для: иконок Browse/Favorites на Sidebar; узлов дерева и стрелок expand/collapse; Top Menu (Home/Import file/Import text/Refresh/Collapse all/Locate); контекстного меню; звезды Favorites и контролов Context Panel (Back/Forward/Collapse/Expand); табов и кнопки pin; Hamburger (Split right/down); переключателя режимов на Status Bar; строки глобального поиска; панели фильтров и поля поиска. Под каждым кейсом — блок «Селекторы» со ссылкой на логические имена из `selectors.md`.
- **Учётки:** для permission-кейсов (Tree-07, Platform-03, MyStuff-04) — отдельный `storageState` под `DATAGROK_SHARING_LOGIN`. Удобно завести второй проект в `playwright.config.ts` (`name: 'chromium-sharing'`) с собственным `storageState`.
- **Matrix-наборы:** «открыть каждое приложение» и «открыть каждый демо-файл» — в отдельных файлах `apps_matrix.test.ts` и `demo_files_matrix.test.ts` (параметризованные тесты по результату API-запроса списка приложений / файлов).
